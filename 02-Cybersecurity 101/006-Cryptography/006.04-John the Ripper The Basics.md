# John the Ripper: The Basics

## Hashes & Why They Matter

- **Hash = fixed-length fingerprint** of data (e.g., MD5, SHA1, NTLM). Same length regardless of input size.
- **Security property:** easy to compute, **hard to invert** (one-way). Brute-forcing tries candidate inputs until a hash matches.
- **Attack model:** use a **dictionary/wordlist**, hash each word, compare to target → match reveals the original.

## John the Ripper (JtR) & Jumbo John

- **Purpose:** high-speed password/hash cracking (dictionary, single, rules).
- **Jumbo John:** feature-rich community build (adds tools like `zip2john`, `rar2john`, `ssh2john`, `unshadow`).

## Setup Essentials

- **Preinstalled:** AttackBox/Kali usually ship **Jumbo John** (`john` shows help/version).
- **Linux install:** `sudo apt install john` or `sudo dnf install john` (may be “core” only). For full features, **build from source** per official guide.
- **Windows:** download prebuilt **jumbo** zip binaries.
- **Wordlists:** `/usr/share/wordlists/` on Kali/AttackBox. Use **rockyou.txt** (from SecLists if needed).

## Core Usage Patterns

**Syntax:** `john [options] [hash_file]`

- **Automatic (quick try):**
    
    `john --wordlist=/path/to/wordlist hash.txt`
    
- **Identify hash type (if auto fails):**
    - Online/`hash-identifier` → suggests formats.
- **Format-specific (faster/safer):**
    
    `john --format=<format> --wordlist=... hash.txt`
    
    *Tip:* for simple “raw” digests use `raw-<algo>` (e.g., `raw-md5`). List formats: `john --list=formats | grep -i md5`.
    

## Windows Auth Hashes (NT/NTLM)

- **Where from:** SAM / `NTDS.dit` dumps (Mimikatz, etc.).
- **Tactics:** sometimes **pass-the-hash**; otherwise crack with JtR using the correct NTLM format.

## Linux `/etc/shadow` Cracking

- **Combine files for John:**
    
    `unshadow /path/passwd /path/shadow > unshadowed.txt`
    
- **Crack (often sha*crypt):**
    
    `john --wordlist=rockyou.txt --format=sha512crypt unshadowed.txt`
    

## Single Crack Mode (smart guesses)

- **Idea:** **mangle** usernames/metadata into likely passwords (e.g., `Markus1!`, `MArkus`, etc.).
- **GECOS support:** uses full name, office, etc., from `/etc/passwd` to enrich guesses.
- **Command:**
    
    `john --single --format=<format> hashes.txt`
    
- **File format requirement:** prefix hash with username → `mike:<hash>`

## Custom Rules (targeted mangling)

- **Why:** encode common org patterns (e.g., Capitalize + digit + symbol).
- **Where:** `john.conf` (`/etc/john/john.conf` or `/opt/john/john.conf`).
- **Building blocks:**
    - `Az` append, `A0` prepend, `c` capitalize; char classes in `[]` (e.g., `[0-9]`, `[!£$%@]`).
- **Example rule:**
    
    ```
    [List.Rules:PoloPassword]
    c Az"[0-9][!£$%@]"
    ```
    
- **Use it:**
    
    `john --wordlist=rockyou.txt --rule=PoloPassword hash.txt`
    

## Archives & Keys

- **Zip:**
    
    `zip2john file.zip > zip_hash.txt` → `john --wordlist=rockyou.txt zip_hash.txt`
    
- **RAR:**
    
    `rar2john file.rar > rar_hash.txt` → `john --wordlist=rockyou.txt rar_hash.txt`
    
- **SSH private keys (`id_rsa`):**
    
    `ssh2john id_rsa > id_rsa_hash.txt` *(or `python3 /opt/john/ssh2john.py ...`)* →
    
    `john --wordlist=rockyou.txt id_rsa_hash.txt`
    

## Practical Takeaways

- Start with **right format** + **right wordlist**; auto-detect is convenient but not always accurate.
- **rockyou.txt** is a strong baseline; combine with **Single** mode and **custom rules** for big wins.
- For system hashes, use the **conversion helpers** (`unshadow`, `zip2john`, `rar2john`, `ssh2john`) to feed John data it understands.
- Enforcing “complexity” isn’t enough—**predictable patterns** are crackable; rules exploit that.
