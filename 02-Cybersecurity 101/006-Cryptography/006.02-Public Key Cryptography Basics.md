# Public Key Cryptography Basics

## High-level goals (security properties)

- **Authentication** — confirm the identity of who you’re talking with.
- **Authenticity** — verify a message genuinely comes from the claimed sender.
- **Integrity** — ensure data hasn’t been altered in transit.
- **Confidentiality** — prevent unauthorized parties from reading data.

Cryptography supplies primitives to satisfy these properties; symmetric (private-key) crypto is fast and mainly for confidentiality, public-key (asymmetric) crypto is used for authentication, signatures, key agreement and bootstrapping symmetric keys.

## Common pattern: asymmetric to negotiate symmetric

- Use asymmetric crypto once to securely exchange/establish a symmetric key (e.g., lock-box analogy: use recipient’s public key to protect a symmetric key), then use fast symmetric ciphers for the session.

## RSA — essentials

- **Security basis:** difficulty of factoring large semiprimes (product of two large primes).
- **Key components / notation:**
    - `p, q` = primes, `n = p·q`.
    - Public key: `(n, e)`; private key: `(n, d)`.
    - `m` = plaintext, `c` = ciphertext.
- **Uses:** encryption, digital signatures, key transport, authentication.
- **CTF / tooling:** common to encounter RSA problems; useful tools: `RsaCtfTool`, `rsatool`.
- **Practical note:** real RSA uses extremely large primes (hundreds of digits); toy examples illustrate math but are insecure.

## Diffie-Hellman (DH) key exchange — essentials

- **Goal:** two parties derive a shared secret over an insecure channel without prior shared secrets.
- **Core idea:** each party combines their private secret with public parameters and the other’s public value; because of the commutative property the resulting shared key matches on both sides.
- **Role in protocols:** DH (or ECDH) for key agreement; combined with signatures (e.g., RSA/ECDSA/Ed25519) for authentication to prevent MITM.

## SSH — applied public-key usage

- **Server authentication:** SSH client checks server public key fingerprint; first-time prompts user to accept and stores the key (`known_hosts`) to detect MITM later.
- **Client authentication:** public-key auth preferred over passwords; client proves identity using private key.
- **Key generation:** `ssh-keygen -t <algorithm>` (common algorithms: `rsa`, `ed25519`, `ecdsa`, `-sk` variants for hardware keys).
- **Key handling best practices:**
    - Keep private keys private, protect with a strong passphrase.
    - Generate keys locally; copy only the public key to servers (e.g., `ssh-copy-id`).
    - Set private key file permissions to `600`.
    - Use `ssh -i <privateKeyFile> user@host` to specify a key.
- **Operational note:** leaving a public key in `~/.ssh/authorized_keys` is effectively a persistent backdoor.

## Digital signatures & certificates

- **Digital signatures:** sign with private key, verify with public key — provides authenticity and integrity. Typical flow: sign hash of document → verifier checks signature and recomputes hash.
- **Certificates (X.509 / TLS):** bind a public key to an identity via a CA chain of trust. Browsers/OSes ship trusted root CAs; a server certificate signed (directly or indirectly) by a trusted CA allows clients to trust the server identity.
- **Practical:** use certificates (Let’s Encrypt or commercial CAs) to enable HTTPS.

## PGP/GPG — email/file encryption & signing

- **PGP/GnuPG:** implement OpenPGP standard for encrypting files, email, and for signing.
- **Typical workflow:** generate keypair (`gpg --full-gen-key`), share public key, encrypt to someone’s public key, decrypt with your private key.
- **Key management:** export/import backups (`gpg --import backup.key`), keys may have expiries and subkeys (signing/encryption).
- **Security:** protect private keys with strong passphrases; `gpg2john` + John the Ripper can be used to attempt passphrase recovery in forensic/CTF settings.

## Practical commands & quick references

- Generate SSH key (Ed25519): `ssh-keygen -t ed25519`
- Use a specific private key: `ssh -i /path/to/private user@host`
- Copy public key to remote: `ssh-copy-id user@host`
- Generate GPG key: `gpg --full-gen-key`
- Import GPG key backup: `gpg --import backup.key`
- Decrypt GPG message: `gpg --decrypt file.gpg`

## Key takeaways / best practices

- Use asymmetric crypto to bootstrap secure channels, then use symmetric ciphers for bulk transport.
- Protect private keys (never share); use passphrases and hardware security keys when possible.
- Verify server fingerprints and certificate chains to prevent MITM.
- Prefer public-key auth for SSH; prefer TLS (certs) for web.
- Back up private keys securely and rotate/expire keys as appropriate.
