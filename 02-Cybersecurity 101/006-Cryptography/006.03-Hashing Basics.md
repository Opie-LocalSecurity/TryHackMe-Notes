# Hashing Basics

## Core ideas

- **Hash value / function:** One-way function maps arbitrary input â†’ fixed-length digest (e.g., MD5, SHA-1, SHA-256). Tiny input changes â†’ large output changes (avalanche).
- **Output encodings:** Typically hex or Base64 (tools like `md5sum`, `sha1sum`, `sha256sum` print hex).

## Why hashing matters

- **Integrity:** Verify files/downloads (same input â‡’ same digest). Great for duplicate detection.
- **Authentication:** Store **hashes** of passwords, not plaintext; server compares hashes at login.

## Collisions

- **Definition:** Different inputs with the same digest (pigeonhole principle).
- **Security note:** **MD5** and **SHA-1** are collision-broken â†’ **do not** use for password storage or new integrity schemes.

## Insecure password storage patterns

- **Plaintext passwords:** e.g., RockYou breach â€” catastrophic reuse risk.
- **Encrypting passwords:** Requires a key; if leaked, all passwords are recoverable.
- **Weak/unsalted hashes:** e.g., LinkedIn used SHA-1 without salt â†’ trivial with wordlists/rainbow tables.

## Defenses for password storage

- **Use slow, memory-hard password hash/KDFs:** **Argon2**, **scrypt**, **bcrypt**, **PBKDF2**.
- **Salt every password (unique per user):** Prevents rainbow tables & identical hashes for identical passwords.
- **Typical flow:**
    1. Choose KDF (e.g., Argon2).
    2. Generate unique random salt.
    3. Compute KDF(password âˆ¥ salt + parameters).
    4. Store **hash + salt + parameters** (not the password).

## Cracking approaches

- **You canâ€™t decrypt a hash.** You must **guess** inputs and hash them:
    - **Tools:** **Hashcat**, **John the Ripper**.
    - **Wordlists:** e.g., `rockyou.txt`; augment with rules/masks as needed.
    - **Hashcat syntax:** `hashcat -m <type> -a <mode> hashfile wordlist` (e.g., `m 3200` for bcrypt, `a 0` straight).
- **GPUs:** Massive speedups for fast hashes; KDFs like bcrypt/scrypt/Argon2 are designed to resist GPU advantage.
- **VM caveats:** VMs often lack direct GPU; prefer host for Hashcat. John uses CPU by default; still faster on host.

## Hashing for integrity checking

- **File verification:** Publishers provide checksums (e.g., SHA-256). If your `sha256sum` matches the official value, the file is identical.
- **Signed checksum files:** Combine hashing with signatures (PGP) for authenticity + integrity.

## Donâ€™t confuse: hashing vs encoding vs encryption

- **Hashing:** One-way digest (integrity, password verification).
- **Encoding (e.g., Base64, UTF-8):** Reversible representation change; **no** secrecy.
- **Encryption:** Reversible with key; ensures **confidentiality**.

- Wordlists: `/usr/share/wordlists/rockyou.txt`

## Key takeaways

- Never store passwords in plaintext or with fast/legacy hashes.
- Always **salt** and use **Argon2/scrypt/bcrypt/PBKDF2** with proper parameters.
- Use hashing to verify file integrity; prefer signed checksum files for authenticity.
- HMACs add **authenticity** to hashing via a shared secret.
