# Networking Core Protocols

## DNS (Layer 7)

- Purpose: Translate names ↔ IPs.
- Ports: UDP/53 by default; TCP/53 as fallback.
- Key records:
    - **A** → hostname → IPv4 (e.g., `example.com` → `93.184.215.14`)
    - **AAAA** → hostname → IPv6
    - **CNAME** → alias one name to another (e.g., `www.example.com` → `example.com`)
    - **MX** → mail exchanger for a domain
- Typical flows:
    - Web browsing: resolver queries **A/AAAA**.
    - Sending email: MTA queries **MX**.
- Tools:
    - `nslookup www.example.com` → returns A/AAAA.
    - Packet view (e.g., `tshark`): separate queries/responses for A then AAAA.

## WHOIS (Domain ownership/metadata)

- Purpose: Shows registrant/registrar data, creation/expiry, updates.
- Requirement: Accurate contact info; privacy proxies can mask it.
- Tools:
    - `whois example.com` (Linux) → registrar, dates, contact (or privacy redaction).

## HTTP / HTTPS (Web)

- Purpose: Browser ↔ web server communication over TCP.
- Ports: **80** (HTTP), **443** (HTTPS); sometimes **8080/8443**.
- Core methods:
    - `GET` (read), `POST` (create/submit), `PUT` (create/replace), `DELETE` (remove).
- Low-level troubleshooting:
    - `telnet HOST 80` then:
        
        ```
        GET /file.html HTTP/1.1
        Host: example
        ```
        
- Observations via Wireshark:
    - Request/response headers reveal metadata (server, last-modified, etc.).

## FTP (File Transfer)

- Purpose: Efficient file transfers.
- Control port: **21/TCP**; data uses a **separate connection**.
- Common commands:
    - `USER`, `PASS`, `LIST`/`RETR` (download), `STOR` (upload), transfer mode (`type ascii|binary`).
- Example flow:
    - `ftp MACHINE_IP` → login (often `anonymous`), `ls`, `type ascii`, `get coffee.txt`.
- Packet insight:
    - Client `ls` translates to server `LIST`; directory listing/file transfer travel over distinct data connections.

## SMTP (Send mail)

- Purpose: Client → mail server and server → server mail transfer.
- Port: **25/TCP** (plain SMTP).
- Core commands:
    - `HELO`/`EHLO`, `MAIL FROM:`, `RCPT TO:`, `DATA` … `.` (end of message), `QUIT`.
- Example via `telnet MACHINE_IP 25`:
    - Issue the sequence above to send a simple message.

## POP3 (Retrieve mail — download model)

- Purpose: Client fetches mail (often deletes from server).
- Port: **110/TCP**.
- Commands:
    - `USER`, `PASS`, `STAT`, `LIST`, `RETR <n>`, `DELE <n>`, `QUIT`.
- Example via `telnet MACHINE_IP 110`:
    - Auth → `STAT`/`LIST` → `RETR 3` to read → `QUIT`.
- Note: Credentials/content are visible in cleartext on the wire.

## IMAP (Synchronize mail — multi-device)

- Purpose: Server-stored mailbox with state sync (read/move/delete).
- Port: **143/TCP**.
- Commands (tagged):
    - `LOGIN <user> <pass>`, `SELECT inbox`, `FETCH 3 body[]`, `MOVE`/`COPY`, `LOGOUT`.
- Example via `telnet MACHINE_IP 143`:
    - `LOGIN` → `SELECT inbox` → `FETCH` message → `LOGOUT`.

## Security & Practical Notes

- Plain **FTP/SMTP/POP3/IMAP** shown here are **unencrypted**; use TLS variants in practice:
    - FTPS/SFTP (protocols differ), **SMTPS** (STARTTLS or 465), **POP3S** (995), **IMAPS** (993).
- DNS queries are typically plaintext; DNS over TLS (DoT) or HTTPS (DoH) can encrypt queries.
- Use packet tools (Wireshark/tshark) to visualize protocol handshakes, commands, and exposures.

## Quick Port Reference

- DNS: 53/UDP (TCP fallback)
- HTTP/HTTPS: 80/443 TCP (also 8080/8443)
- FTP: 21 TCP (plus separate data connection)
- SMTP: 25 TCP
- POP3: 110 TCP
- IMAP: 143 TCP

## Handy CLI Snippets

- DNS: `nslookup host` | Packet view: `tshark -r file.pcapng`
- WHOIS: `whois domain`
- HTTP test: `telnet host 80` then `GET / HTTP/1.1` + `Host: x`
- FTP: `ftp host` → `USER anonymous`
- SMTP: `telnet host 25` → `HELO …`
- POP3: `telnet host 110` → `USER/PASS`
- IMAP: `telnet host 143` → `LOGIN/SELECT/FETCH`
