# Firewall Fundamentals

## 🔒 **Purpose of a Firewall**

- Acts as a **security gatekeeper** between a private network/device and external networks (like the Internet).
- **Inspects** all incoming and outgoing traffic.
- **Applies rules** to allow or block specific traffic based on defined policies.
- Prevents **unauthorized access** and potential cyber threats.

## 🧱 **Types of Firewalls**

| **Type** | **OSI Layer(s)** | **Key Features** | **Limitations / Notes** |
| --- | --- | --- | --- |
| **Stateless Firewall** | Layer 3–4 | - Filters based on static rules- Fast processing | - No memory of past connections- Limited security context |
| **Stateful Firewall** | Layer 3–4 | - Tracks connection states (via a **state table**)- Allows or blocks packets based on session history | - More resource-intensive than stateless |
| **Proxy Firewall** (Application-level) | Layer 7 | - Inspects packet content- Masks internal IPs (anonymity)- Supports content filtering | - Slower due to deep inspection |
| **Next-Generation Firewall (NGFW)** | Layers 3–7 | - Deep packet inspection- Intrusion prevention system (IPS)- SSL/TLS decryption- Heuristic analysis for threats | - Most advanced; higher cost and complexity |

## ⚙️ **Firewall Rules**

**Purpose:** Define how specific types of traffic are handled.

**Main Components:**

- **Source address** – IP initiating traffic
- **Destination address** – IP receiving traffic
- **Protocol** – e.g., TCP, UDP
- **Port** – e.g., 22 (SSH), 80 (HTTP)
- **Action** – What to do (Allow, Deny, Forward)
- **Direction** – Whether rule applies to inbound, outbound, or forwarded traffic

### **Actions**

| **Action** | **Purpose** | **Example** |
| --- | --- | --- |
| **Allow** | Permit traffic matching rule | Allow outbound TCP port 80 (HTTP) |
| **Deny** | Block specified traffic | Deny inbound TCP port 22 (SSH) |
| **Forward** | Redirect traffic internally | Forward inbound TCP port 80 → 192.168.1.8 (web server) |

### **Directionality**

| **Direction** | **Purpose** | **Example** |
| --- | --- | --- |
| **Inbound** | Rules for incoming packets | Allow HTTP to web server |
| **Outbound** | Rules for outgoing packets | Block all SMTP except from mail server |
| **Forward** | For internal routing | Forward web traffic to local web host |

## 🪟 **Windows Defender Firewall**

### **Overview**

- Built into Windows OS; manages inbound/outbound traffic.
- Accessible via **Control Panel → Windows Defender Firewall**.
- Default: **Enabled** for all network profiles.

### **Network Profiles**

1. **Private:** Home or trusted networks.
2. **Public:** Untrusted networks (e.g., cafés); stricter blocking.

### **Key Functions**

- **Allow/Block applications** per profile.
- **Restore defaults** easily.
- **Custom Rules** through *Advanced Settings*.

### **Example: Creating a Custom Rule**

**Goal:** Block all outbound HTTP (port 80) and HTTPS (port 443).

Steps:

1. Open **Advanced Settings → Outbound Rules → New Rule**.
2. Choose **Custom**, select **TCP**, specify ports `80,443`.
3. Action: **Block the connection**.
4. Apply to all profiles → Name and finish.
5. Test: Browsing HTTP/HTTPS sites fails (rule works).

## 🐧 **Linux Firewalls**

### **Netfilter Framework**

- Core Linux kernel module for **packet filtering**, **NAT**, and **connection tracking**.
- Used by tools like:
    - **iptables** – Traditional CLI utility.
    - **nftables** – Successor with simplified syntax.
    - **firewalld** – Zone-based, dynamic configuration.

### **ufw (Uncomplicated Firewall)**

- Simplified front-end for `iptables`.
- Easy command-line syntax.

**Basic Commands:**

```bash
sudo ufw status          # Check status
sudo ufw enable          # Enable firewall
sudo ufw disable         # Disable firewall
sudo ufw default allow outgoing  # Allow all outgoing by default
sudo ufw deny 22/tcp     # Block incoming SSH
sudo ufw status numbered # Show active rules
sudo ufw delete <num>    # Remove rule by number
```

**Example:**

Block SSH traffic:

```bash
sudo ufw deny 22/tcp
```

## **Key Takeaway**

Firewalls act as the **first line of defense** in network security, inspecting and controlling traffic through **rule-based**, **stateful**, or **content-aware** mechanisms, protecting systems from unauthorized or malicious network activity.
