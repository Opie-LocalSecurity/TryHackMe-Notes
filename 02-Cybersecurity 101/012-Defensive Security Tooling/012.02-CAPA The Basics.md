# CAPA: The Basics

## 🔍 **Overview of CAPA**

**Purpose:**

- CAPA (Common Analysis Platform for Artifacts), developed by FireEye Mandiant, automates *static analysis* of executables (PE, ELF, .NET, shellcode).
- Identifies **capabilities** (what a program can do) by applying rule-based pattern matching.
- Maps findings to **MITRE ATT&CK**, **MAEC**, and **MBC** frameworks.
- Enables malware analysts to understand software functionality without manual reverse engineering.

## ⚙️ **Running CAPA**

**Basic usage (PowerShell):**

```powershell
capa.exe .\cryptbot.bin
```

**Common options:**

| Option | Description |
| --- | --- |
| `-h` | Show help |
| `-v` | Verbose output |
| `-vv` | Very verbose output |

**Example Output Includes:**

- **File metadata:** hashes, OS, format, architecture
- **ATT&CK mapping:** tactics & techniques
- **MAEC category:** malware behaviour class (e.g., launcher, downloader)
- **MBC behaviours:** granular malware actions (anti-analysis, file ops, encoding, etc.)
- **Capabilities & namespaces:** rule-based detections grouped by behaviour type

## 🧩 **Dissecting CAPA Results**

### **General Information**

- **md5/sha1/sha256:** file hashes
- **analysis:** static or dynamic
- **os, arch, format:** binary context
- **path:** file location

### **MITRE ATT&CK Integration**

- Framework describing adversary tactics and techniques.
- CAPA maps behaviours to ATT&CK identifiers (e.g., `Defense Evasion::Obfuscated Files [T1027]`).
- Helps analysts link malware behaviour to known attack patterns.

### **MAEC (Malware Attribute Enumeration and Characterization)**

- Describes **malware type** or high-level purpose:
    - **Launcher:** triggers or loads malicious code.
    - **Downloader:** fetches and executes additional payloads.

Example:

```
malware-category | launcher
```

### **MBC (Malware Behavior Catalogue)**

**Purpose:**

Catalogues objectives, behaviours, and methods derived from ATT&CK and malware traits.

**Structure:**

`Objective::Behavior::Method [Identifier]`

**Example:**

```
DATA::Encode Data::Base64 [C0026.001]
```

**Key Concepts:**

- **Objective:** high-level intent (Execution, Persistence, Defense Evasion, etc.)
- **Behavior:** what the malware does (e.g., Encode Data, Create Process)
- **Method:** how it’s done (e.g., Base64 encoding, XOR)
- **Micro-behavior:** low-level actions (read/write files, allocate memory)

### **Namespaces**

**Definition:**

Organizes related rules and capabilities into functional groups.

**Format:**

`Capability :: Top-Level Namespace / Namespace`

**Example:**

```
reference Base64 string :: data-manipulation/encoding/base64
```

**Main Top-Level Namespaces:**

| TLN | Purpose |
| --- | --- |
| anti-analysis | Detects anti-debugging, VM evasion |
| communication | Network activity (HTTP, C2) |
| data-manipulation | Encoding/encryption behaviours |
| host-interaction | File/process operations |
| impact | Damage or exfiltration actions |
| persistence | Maintaining system access |
| load-code | Code injection/loading |
| linking | Dynamic linking/loading |
| executable | PE structure, TLS sections |
| nursery | Staging for experimental rules |

Each namespace contains multiple **rule YAML files** defining detection patterns (e.g., regex, strings, API calls).

### **Capabilities**

**Definition:**

Specific rules that describe observed program abilities.

**Example Breakdown:**

| Field | Example | Explanation |
| --- | --- | --- |
| Capability | reference Base64 string | Malware encodes data using Base64 |
| TLN | data-manipulation | Related to data transformation |
| Namespace | encoding/base64 | Specific encoding rule group |
| Rule File | reference-base64-string.yml | YAML rule applied |

**Other Examples:**

- `reference anti-VM strings` → detects VM evasion
- `schedule task via schtasks` → identifies persistence via Windows tasks

## 🧠 **Deep Analysis with `vv` and JSON Output**

- Use `vv` for **detailed rule match conditions**.
- Use `j` to export to JSON:
    
    ```powershell
    capa.exe -j -vv .\cryptbot.bin > cryptbot_vv.json
    ```
    
- The JSON file can be explored visually using **CAPA Web Explorer**.

## 🌐 **CAPA Web Explorer**

**Purpose:** Interactive UI for CAPA results.

- Load `.json` output locally or online via [Mandiant CAPA Explorer](https://mandiant.github.io/capa/explorer/#/).
- Features:
    - Rule browsing and filtering
    - Global search
    - Inspection of matched strings, regexes, and logic from YAML rules

**Example Rules:**

- *Anti-VM Detection:* regex for `/VMWare/i`, `/vmtoolsd.exe/i`
- *Scheduled Task Persistence:* looks for strings `/schtasks/i` and `/\/create /i`

## 🧾 **Summary of CAPA Analysis Workflow**

1. **Run CAPA** on a binary (`capa.exe <file>`).
2. **Review metadata** and high-level ATT&CK/MAEC mapping.
3. **Analyze detailed behaviours** via MBC and capabilities.
4. **Inspect namespaces** to group related functionality.
5. **Export results** as JSON with `vv -j`.
6. **Explore visually** using CAPA Web Explorer to see rule triggers and conditions.

## 🧭 **Key Takeaways**

- CAPA automates expert-level static analysis using community-curated rules.
- Integrates with **ATT&CK**, **MAEC**, and **MBC** for context-rich insights.
- Rules and namespaces make malware analysis standardized and scalable.
- CAPA Web Explorer transforms raw data into an interactive, human-readable format.
