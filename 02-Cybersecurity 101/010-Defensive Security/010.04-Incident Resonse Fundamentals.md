# Incident Resonse Fundamentals

## 🧭 **Introduction to Incident Response**

- **Purpose:** Manage cybersecurity incidents from detection to resolution—minimizing damage and recovery time.
- **Analogy:** Like securing a home with guards, alarms, and plans for both prevention and reaction after a break-in.
- **Goal:** Prepare, detect, respond, and learn from incidents to strengthen future defenses.

## ⚠️ **What Are Incidents?**

- **Events:** Every system process generates logs (events). Some events may indicate malicious activity.
- **Security Solutions:** Tools analyze logs and raise **alerts** when suspicious patterns appear.
- **Alert Types:**
    - **False Positive:** Benign activity flagged as malicious (e.g., backup traffic mistaken for data exfiltration).
    - **True Positive:** Verified harmful activity (e.g., real phishing email).
- **Incident Severity Levels:**
    - **Low → Medium → High → Critical**, based on potential impact and urgency.

## 🧨 **Types of Incidents**

1. **Malware Infections:** Malicious software damaging systems or data.
2. **Security Breaches:** Unauthorized access to confidential data.
3. **Data Leaks:** Exposure of sensitive data—can be accidental or deliberate.
4. **Insider Attacks:** Malicious actions from within the organization.
5. **Denial of Service (DoS):** Overwhelming a system to make it unavailable.

*Impact depends on the organization’s nature—what’s critical for one may be minor for another.*

## 🔁 **Incident Response Process**

### **SANS Framework (PICERL – 6 Phases)**

| Phase | Purpose | Example |
| --- | --- | --- |
| **Preparation** | Build teams, plans, and defenses. | Employee phishing awareness training. |
| **Identification** | Detect abnormal behavior. | Suspicious data transfer detected. |
| **Containment** | Limit damage and spread. | Isolate infected system. |
| **Eradication** | Remove root cause or threat. | Run malware cleanup. |
| **Recovery** | Restore and validate systems. | Rebuild host, restore data. |
| **Lessons Learned** | Review and improve processes. | Post-incident review meeting. |

### **NIST Framework (4 Phases)**

- **Preparation**
- **Detection & Analysis**
- **Containment, Eradication, & Recovery**
- **Post-Incident Activity**

**Both frameworks** guide structured and repeatable response handling.

## 🧾 **Incident Response Plan (IRP)**

- A formal, approved document describing:
    - Roles and responsibilities
    - Response methodology
    - Communication and escalation procedures
    - Stakeholder and law enforcement contact plans

## 🧠 **Incident Response Techniques**

Security tools automate detection and help execute response actions:

| Tool | Purpose |
| --- | --- |
| **SIEM (Security Information and Event Management)** | Centralizes and correlates logs to identify incidents. |
| **AV (Antivirus)** | Detects and removes known malware. |
| **EDR (Endpoint Detection & Response)** | Protects endpoints, detects advanced threats, enables containment and remediation. |

## 📚 **Playbooks vs. Runbooks**

- **Playbooks:** High-level response guides for specific incident types (e.g., phishing, malware, DoS).
    
    **Example – Phishing Email Playbook:**
    
    1. Notify stakeholders.
    2. Analyze email headers/body.
    3. Check attachments.
    4. Identify users who opened it.
    5. Isolate infected systems.
    6. Block sender.
- **Runbooks:** Step-by-step, tool-specific technical instructions for executing parts of a playbook.

## ✅ **Key Takeaways**

- Incident response = **structured, proactive, and reactive** defense strategy.
- Frameworks (SANS/NIST) ensure consistency.
- Tools (SIEM, AV, EDR) aid detection and containment.
- Documentation (Plans, Playbooks, Runbooks) ensures clarity, accountability, and rapid action.
