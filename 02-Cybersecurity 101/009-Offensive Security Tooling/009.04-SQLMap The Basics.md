# SQLMap: The Basics

## What a database + SQL are

- Database = structured storage for application data (users, products, logs…).
- DBMS (MySQL, PostgreSQL, SQLite, MSSQL, etc.) accepts SQL (Structured Query Language) to read/write data.
- Web apps issue SQL queries from user inputs (search boxes, login forms, etc.) to the DBMS.

## Core idea of SQL injection (SQLi)

- SQLi happens when user input is **not properly validated/escaped** and becomes part of an executable SQL statement.
- An attacker can craft input that changes the intended query logic (e.g., inject `OR 1=1`) to bypass checks or extract data.
- Always test only with explicit owner permission.

## Minimal example (login bypass)

- Intended query:
    
    ```
    SELECT * FROM users WHERE username = 'John' AND password = 'Un@detectable444';
    ```
    
- Malicious password input:
    
    `abc' OR 1=1;-- -`
    
    produces (after concatenation):
    
    ```
    SELECT * FROM users WHERE username = 'John' AND password = 'abc' OR 1=1;-- -';
    ```
    
    `OR 1=1` makes the WHERE true regardless of password; `-- -` comments out the rest.
    

## Common SQLi techniques (identified in example)

- **Boolean-based blind** — change query to evaluate true/false and infer data from responses.
- **Error-based** — craft payloads that produce DB errors revealing information.
- **Time-based blind** — use `SLEEP()` to infer true/false via response time.
- **UNION-based** — use `UNION SELECT` to retrieve data by aligning columns.

## sqlmap — automated exploitation tool (practical usage)

- `sqlmap --wizard` — interactive guided mode for beginners.
- Basic flow:
    - Test target: `sqlmap -u 'http://target/search?cat=1'`
    - Enumerate DBs: `-dbs`
    - List tables: `D <database> --tables`
    - Dump table contents: `D <db> -T <table> --dump`
    - Use `r <request_file>` to supply an intercepted POST request.
- Helpful flags/options mentioned:
    - `-level=5` — in-depth scanning (slower, more tests).
    - Respond to prompts as required (tool may ask to restrict DBMS tests, try union char changes, etc.).
- sqlmap will report injection points, payload types, and back-end info (DBMS, webserver, app tech).

## GET vs POST testing notes

- GET parameters visible in URL are easy to test directly.
- If form parameters are sent via GET but not visible, open browser DevTools → Network tab → submit form to view full request/URL.
- For POST-based forms, intercept and save the full HTTP request then use `sqlmap -r intercepted_request.txt`.

## Practical lab specifics (from exercise)

- Vulnerable login page: `http://MACHINE_IP/ai/login` → request endpoint visible as:
    
    `http://MACHINE_IP/ai/includes/user_login?email=test&password=test` (copy and wrap in single quotes when using in shell: `'http://...?...'`).
    
- If simple scan is insufficient, add `-level=5`.
- Follow recommended answers in the lab when sqlmap asks interactive questions (examples given in text).

## Outcomes & takeaways

- SQL injection allows authentication bypass and full data extraction when inputs are unsanitized.
- Automation (sqlmap) speeds discovery and exploitation but requires careful option selection and sometimes deeper testing (`-level`).
- Defensive reminder: sanitize/parameterize queries (prepared statements / ORM parameter binding), least privilege DB accounts, WAFs, input validation, and secure error handling.
