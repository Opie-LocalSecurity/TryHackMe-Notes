# Security Principles

## **CIA Triad (Core Security Principles)**

- **Confidentiality** – Only authorized users can access data.
- **Integrity** – Data must remain accurate and unaltered; any change should be detectable.
- **Availability** – Systems and data must be accessible when needed.

**Examples:**

- *Online shopping:* Protect card data (Confidentiality), prevent order tampering (Integrity), keep site running (Availability).
- *Healthcare:* Secure patient data (Confidentiality), ensure medical records are accurate (Integrity), and system uptime (Availability).

## **Beyond CIA**

- **Authenticity** – Ensures data or documents come from a legitimate source.
- **Nonrepudiation** – Prevents the sender from denying the creation or transmission of data.
    
    → Both are vital for accountability in systems such as banking and e-commerce.
    

## **Parkerian Hexad (Expanded Model)**

1. **Availability**
2. **Utility** – Information must remain useful (e.g., encrypted data without a key = no utility).
3. **Integrity**
4. **Authenticity**
5. **Confidentiality**
6. **Possession** – Prevent unauthorized control or theft of data (e.g., ransomware, stolen drives).

## **DAD Triad (Opposite of CIA)**

- **Disclosure** ↔ attacks Confidentiality
- **Alteration** ↔ attacks Integrity
- **Destruction/Denial** ↔ attacks Availability

→ Example: In healthcare, leaked patient data (Disclosure), tampered records (Alteration), or unavailable systems (Destruction).

**Security balance:** Overemphasizing one aspect (e.g., confidentiality) can harm others (e.g., availability).

## **Security Models**

### **Bell-LaPadula (Confidentiality)**

- *Simple Security Property:* “No read up”
- *Star Security Property:* “No write down”
- *Discretionary Security Property:* Access matrix controls read/write.
    
    → Summary: **Write up, Read down**
    

### **Biba (Integrity)**

- *Simple Integrity Property:* “No read down”
- *Star Integrity Property:* “No write up”
    
    → Summary: **Read up, Write down**
    

### **Clark-Wilson (Integrity)**

- **CDI:** Constrained data items (protected data).
- **UDI:** Unconstrained data (e.g., inputs).
- **TPs:** Transformation Procedures – maintain CDI integrity.
- **IVPs:** Integrity Verification Procedures – validate data integrity.

Other models: Brewer–Nash, Goguen–Meseguer, Sutherland, Graham–Denning, Harrison–Ruzzo–Ullman.

## **Defence-in-Depth**

- Multiple security layers reduce the chance of a total breach (e.g., building locks, access control, surveillance).
- Aim: Slow attackers and increase overall resilience.

## **ISO/IEC 19249:2017 – Security Architecture & Design Principles**

### **Architectural Principles**

1. **Domain Separation** – Isolate processes/resources (e.g., user mode vs. kernel mode).
2. **Layering** – Organize systems in layers (e.g., OSI model) for better control.
3. **Encapsulation** – Hide internal logic; use APIs to prevent misuse.
4. **Redundancy** – Add backups for availability/integrity (e.g., RAID, dual power).
5. **Virtualization** – Isolate environments for security (e.g., VMs, sandboxing).

### **Design Principles**

1. **Least Privilege** – Grant only the permissions required.
2. **Attack Surface Minimization** – Disable unnecessary services to reduce risk.
3. **Centralized Parameter Validation** – Validate inputs via a single mechanism.
4. **Centralized Security Services** – Centralize auth and security controls (avoid single points of failure).
5. **Error & Exception Handling** – Fail safely, avoid leaking sensitive info in errors.

## **Zero Trust vs Trust but Verify**

- **Trust but Verify:** Assume trust but confirm with logs, IDS, IPS, proxies.
- **Zero Trust:** “Never trust, always verify.” Every entity is authenticated and authorized continuously.
    
    → Uses **microsegmentation** (isolating hosts/networks).
    
    → Balancing practicality with strict security is crucial.
    

## **Threat, Vulnerability, and Risk**

- **Vulnerability:** Weakness (e.g., unpatched software).
- **Threat:** Potential danger exploiting the weakness.
- **Risk:** Likelihood and impact of the threat’s exploitation.
    
    → Example: Vulnerable hospital database + exploit code → high risk, urgent action needed.
    

## **Shared Responsibility Model (Cloud Security)**

- **IaaS:** Customer secures OS, apps, and data.
- **SaaS:** Provider handles OS/infrastructure; customer secures usage and access.
    
    → Security success depends on both provider and customer fulfilling their roles.
    

## ✅ **Summary Focus**

- **CIA Triad** = Foundation of security.
- **DAD Triad** = Opposing attack concepts.
- **Security Models** (Bell-LaPadula, Biba, Clark-Wilson) = Guide confidentiality/integrity control.
- **Defence-in-Depth** and **ISO/IEC 19249** = Layered protection & secure design.
- **Zero Trust** = Modern principle for minimizing insider and network trust.
- **Threat/Risk Framework** = Assess likelihood and impact.
- **Shared Responsibility** = Collaboration between cloud providers and users for end-to-end security.
