# OWASP Top 10 2025: Application Design Flaws

## **AS02: Security Misconfigurations**

### **Core Idea**

Failures in configuration (not code) that leave systems exposed through unsafe defaults, open services, or incomplete settings.

### **Impact**

- Creates easy attack surfaces
- Can expose sensitive data or allow privilege escalation
- Often affects cloud services, containers, admin panels, and network access

### **Common Patterns**

- Default/weak credentials
- Exposed services, endpoints, or admin panels
- Public cloud storage (S3, Azure Blob, GCP buckets)
- Unrestricted APIs or missing auth
- Verbose error messages with stack traces
- Outdated software or containers
- Exposed AI/ML endpoints without access controls

### **Prevention**

- Harden defaults; disable unused features
- Strong auth + least privilege
- Network segmentation and restricted exposure
- Patch software and containers
- Hide stack traces/system info
- Regular cloud config audits
- Secure AI endpoints
- Integrate automated config checks into CI/CD

## **AS03: Software Supply Chain Failures**

### **Core Idea**

Security risks from compromised, outdated, or unverified components—libraries, services, build tools, or AI models.

### **Impact**

- One compromised dependency can compromise the entire system
- Attacks propagate automatically via updates or packages
- Difficult to detect; affect both code and AI artefacts

### **Common Patterns**

- Unverified/unmaintained libraries
- Auto-updates without integrity checks
- Blind trust in third-party AI models
- Weak CI/CD or build pipeline security
- Poor provenance/license tracking
- No post-deployment dependency monitoring

### **Protections**

- Verify all dependencies and AI models
- Continuous vulnerability monitoring
- Sign and verify updates/packages
- Lock down CI/CD pipelines
- Track provenance/licensing
- Runtime monitoring for dependency/AI anomalies
- Include supply chain threat modelling in SDLC

## **AS04: Cryptographic Failures**

### **Core Idea**

Weak, missing, or misused encryption—poor key handling, weak algorithms, or unencrypted data.

### **Impact**

- Exposure of sensitive data (passwords, tokens, PII)
- MITM attacks, brute-force attacks on weak keys
- Secrets leaked via code, configs, or AI systems

### **Common Patterns**

- Deprecated algorithms (MD5, SHA-1, ECB mode)
- Hard-coded secrets
- Weak key management or rotation
- Missing encryption at rest/in transit
- Invalid/self-signed TLS certs
- AI models handling sensitive data without secure secret management

### **Prevention**

- Use strong modern crypto (AES-GCM, ChaCha20-Poly1305, TLS 1.3)
- Use dedicated key management services (AWS KMS, Azure Key Vault, Vault)
- Regular key rotation and lifecycle policies
- Maintain inventory of keys/certificates
- Ensure AI systems never expose secrets
- Enforce secure storage and transmission standards

## **AS06: Insecure Design**

### **Core Idea**

System logic or architecture is insecure from the start due to flawed assumptions, missing threat modelling, or weak business logic. In the AI era, this includes unguarded LLMs and unsafe automation.

### **Impact**

- Cannot be patched easily—requires redesign
- Breaks trust boundaries and system workflow
- AI systems amplify risks (prompt injection, unverified outputs, compromised models)

### **Common Insecure Designs**

- Weak business logic (e.g., flawed recovery or approval flows)
- Assumptions about user or AI behaviour
- AI components with excessive authority
- Missing LLM guardrails; prompt injection exposure
- Debug/test bypasses in production
- No AI-specific threat modelling

### **AI-Era Design Risks**

- Prompt injection
- Blind reliance on model output
- Poisoned or unverified models
- Hidden model backdoors or unsafe fine-tuned data
- Automation without human oversight

### **Secure Design Practices**

- Treat all models as untrusted
- Validate and filter AI inputs/outputs
- Separate system prompts from user content
- Avoid placing sensitive data in prompts
- Require human review for high-risk actions
- Track model provenance; monitor behaviour
- Use differential privacy for sensitive data
- Include AI-specific threat modelling across the lifecycle
- Define security requirements before building features
- Enforce least privilege for users, APIs, and services
- Strong authentication/authorisation/session management
- Keep dependencies verified and updated
- Continuous testing for logic flaws and abuse paths
