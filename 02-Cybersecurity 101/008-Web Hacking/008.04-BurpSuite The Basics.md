# BurpSuite: The Basics

## Core idea

- Java-based framework for manual web (and API/mobile) application penetration testing.
- Works by intercepting and letting you view/modify HTTP(S) traffic between client and server — central to manual testing workflows.

## Editions (focus: Community)

- **Community**: free, local/manual testing; useful feature set but rate-limited and cannot save projects.
- **Professional**: full feature set — automated scanner, unrestricted Intruder (fuzzer/brute-forcer), project saving, API, full extension support, Burp Collaborator.
- **Enterprise**: server-based continuous automated scanning (for CI/continuous monitoring).

## Key modules (what you’ll use most)

- **Proxy**: intercept, view, modify, forward/drop requests & responses; logs HTTP & WebSocket history.
- **Repeater**: craft, edit, resend single requests — great for manual payload iteration.
- **Intruder**: parameter fuzzing / brute force (rate-limited in Community).
- **Decoder**: encode/decode and transform data (base64, URL, hex, etc.).
- **Comparer**: byte/word-level diffs between responses or data blobs.
- **Sequencer**: analyze randomness of tokens (session IDs, cookies).
- **Extender / BApp Store**: load/extend Burp via Java, Jython (Python), JRuby (Ruby); many community extensions (e.g., Logger++).

## Dashboard & navigation

- Dashboard shows Tasks, Event log, Issue activity (Pro only), Advisory (Pro report data).
- Top menu selects modules; second menu shows sub-tabs.
- Tabs can be detached into separate windows.
- Useful keyboard shortcuts:
    - `Ctrl+Shift+D` Dashboard
    - `Ctrl+Shift+T` Target
    - `Ctrl+Shift+P` Proxy
    - `Ctrl+Shift+I` Intruder
    - `Ctrl+Shift+R` Repeater

## Settings

- Two types: **User (global)** and **Project** (project-only; Community cannot save projects).
- Settings UI includes search, user/project filter, categories; many tools link directly to relevant settings.

## Proxy specifics & useful features

- Intercept requests/responses (toggle `Intercept is on` to pause/resume).
- Logs everything to **HTTP history** and **WebSockets history** even when intercept is off.
- **Response interception rules**: by default responses aren't intercepted unless rules request it.
- **Match & Replace**: regex-based automated modifications on requests/responses (useful for user-agent, cookies, etc.).
- Option to restrict interception to in-scope URLs (prevent noise).

## Targeting / Scoping / Site map

- **Site map**: automatically builds a tree of visited pages and API endpoints.
- **Scope**: include/exclude domains or IPs to limit recording and testing surface.
- **Issue definitions**: reference list of scanner-detected vulnerabilities (useful even without Pro scanner).
- Best practice: add target to scope and configure Proxy to ignore out-of-scope traffic.

## Burp Browser

- Built-in Chromium configured to use Burp Proxy (convenient; avoids manual browser proxy config).
- On Linux as root, sandbox may fail — options: run under non-root user (recommended) or allow browser without sandbox (security risk).

## Intercepting HTTPS (TLS)

- Burp acts as a TLS man-in-the-middle; browsers will warn unless they trust PortSwigger CA.
- Typical steps to trust Burp CA:
    1. With Proxy on, visit `http://burp/cert` → download `cacert.der`.
    2. Import into browser certificate manager (e.g., Firefox `about:preferences` → View Certificates → Import).
    3. Mark “Trust this CA to identify websites”.
- After import, TLS sites will load without cert errors.

## Example: reflected XSS (practical walkthrough)

- Problem: client-side filtering prevents special chars in email field.
- Approach:
    1. Fill form with legitimate values and submit while Proxy intercept is **on**.
    2. In intercepted request, replace email with payload: `<script>alert("Succ3ssful XSS")</script>`.
    3. URL-encode the payload (`Ctrl+U` in Burp) and forward the request.
    4. If server reflects the payload unescaped into a response, the alert executes — confirms reflected XSS.
- Takeaway: client-side filters are bypassable; intercept and mutate requests server-side.

## Practical checklist / quick tips

- Always set and verify **scope** before active testing to avoid capturing unrelated traffic.
- Import Burp CA into any browser you use with the proxy.
- Use Burp Browser for convenience when possible; prefer running Burp under a non-root account.
- Send interesting requests to **Repeater** for iterative testing, **Comparer** for diffing responses, **Intruder** for parameter fuzzing (mind Community rate limits).
- Use **Match & Replace** for repetitive transformations (fix tokens, headers) during testing.
- Explore the **Extender / BApp Store** to fill gaps (e.g., enhanced logging, custom scanners).
- Remember Community limits: no project save, limited Intruder speed, no automated scanner.
