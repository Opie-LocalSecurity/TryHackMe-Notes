# Javascript Essentials

## Core Concepts

- **Variables**
    - Declarations: `var` (function-scoped), `let` & `const` (block-scoped).
    - Use `let` for reassignable values; `const` for constants; avoid `var` in modern code.
- **Data Types**
    - Primitives: `string`, `number`, `boolean`, `null`, `undefined`, `bigint`, `symbol`.
    - Reference: `object` (includes arrays, functions, plain objects).
- **Functions**
    - Encapsulate reusable logic, e.g., `function PrintResult(rollNum) { ... }`.
    - Can accept parameters and return values; reduce duplication.
- **Loops**
    - `for`, `while`, `do...while` to repeat work until a condition becomes `false`.
    - Typical pattern: iterate arrays and call helper functions.
- **Request–Response Cycle (Web)**
    - Browser (client) sends HTTP request → server responds with HTML/CSS/JS/data.

## JavaScript Overview & Execution

- **Interpreted language**: runs directly in the browser (no pre-compilation).
- **Control flow**: `if/else`, comparisons, and logical operators to branch behavior.
- **DevTools Console (Chrome)**
    - Open with `Ctrl+Shift+I` → **Console** tab.
    - Execute snippets for quick testing (e.g., simple addition, logging).

## Integrating JS with HTML

- **Internal JS**
    - Inline within `<script>` tags in the same HTML file.
    - Good for small demos; can interact with DOM via `document.getElementById(...).innerHTML`.
- **External JS**
    - Separate `.js` file loaded with `<script src="script.js"></script>`.
    - Cleaner organization, reuse, and caching; preferred for real projects.
- **How to tell which is used**
    - **View Source**: inline code = `<script> ... </script>`; external = `<script src="...">`.

## Built-in Dialog APIs & Abuse Risks

- **`alert(msg)`**: info popup (OK).
- **`prompt(msg)`**: asks for input; returns string or `null`.
- **`confirm(msg)`**: yes/no; returns `true`/`false`.
- **Abuse scenario**: loops triggering hundreds of alerts to “lock” the UI (nuisance). Ties into broader XSS risks if untrusted JS executes in a page.

## Control Flow & (Bad) Client-Side Auth

- **Conditional example**: prompt age → `if (age >= 18) ... else ...`
- **Caution**: Implementing **login/auth in client-side JS** is insecure—logic and secrets are visible and tamperable. Server must enforce authentication and authorization.

## Minification, Obfuscation, and Deobfuscation

- **Minification**: removes whitespace/comments, shortens names → smaller, faster loads.
- **Obfuscation**: intentionally scrambles structure to hinder reading (still executable).
- **Practical takeaway**: Obfuscated code still runs; attackers (and defenders) can **deobfuscate** with tools to analyze behavior.

## Security-Focused Best Practices

- **Don’t rely on client-side validation alone**
    
    Always validate on the **server**; JS can be disabled or manipulated.
    
- **Be careful with third-party scripts**
    
    Load only trusted libraries via verified sources; typo-squatted packages are common.
    
- **Never hardcode secrets in frontend**
    
    API keys/tokens/passwords in JS are exposed to users.
    
- **Ship production code minified/obfuscated**
    
    Improves performance and raises the effort bar for reverse-engineering (but is not security by itself).
    

## Mental Model / Outcomes

- You can:
    - Declare variables with correct scope, work with common data types.
    - Write functions and loops to avoid repetition.
    - Run JS in the Chrome console and wire it into HTML (internal/external).
    - Use dialog APIs responsibly and recognize simple abuse patterns.
    - Understand why auth and validation must be server-enforced.
    - Recognize, create, and reason about minified/obfuscated code—and know that deobfuscation is possible.
