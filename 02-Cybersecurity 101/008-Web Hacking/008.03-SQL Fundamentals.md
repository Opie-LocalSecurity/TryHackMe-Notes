# SQL Fundamentals

## Why it matters (security context)

- Databases underpin most security work: auth data, SIEM/SOC stores, malware/threat tooling, and web apps (incl. SQLi on offense; triage/forensics on defense).

## Database types

- **Relational (SQL):** structured, tabular (tables → rows/columns), enforce schema & relationships; great for accuracy/consistency (e-commerce, transactions).
- **Non-relational (NoSQL):** non-tabular (documents, key-value, etc.), flexible schema; great for varied/irregular data (social content, logs).
    
    ```json
    { _id:ObjectId(...), name:{first:"Thomas",last:"Anderson"}, occupation:["The One"], ... }
    ```
    

## Core relational concepts

- **Table:** named set of rows with defined columns (types: INT, VARCHAR, DATE, etc.).
- **Row / Record:** one item (e.g., a book).
- **Column:** attribute with a data type and constraints (e.g., NOT NULL).
- **Primary Key (PK):** unique identifier per row (one per table).
- **Foreign Key (FK):** column(s) referencing a PK in another table, enabling relationships.

## SQL & DBMS

- **DBMS examples:** MySQL, Oracle, MariaDB (relational); MongoDB (non-relational).
- **SQL purpose:** define, query, and manipulate relational data.
- **Benefits:** fast, readable, reliable (schema/constraints), flexible querying.

## Database statements (DDL)

- **Create DB:** `CREATE DATABASE thm_bookmarket_db;`
- **List DBs:** `SHOW DATABASES;`
- **Use DB:** `USE thm_bookmarket_db;`
- **Drop DB:** `DROP DATABASE db_name;` *(destructive)*

## Table statements (DDL)

- **Create table:**
    
    ```sql
    CREATE TABLE book_inventory (
      book_id INT AUTO_INCREMENT PRIMARY KEY,
      book_name VARCHAR(255) NOT NULL,
      publication_date DATE
    );
    ```
    
- **List tables:** `SHOW TABLES;`
- **Inspect schema:** `DESCRIBE book_inventory;` (aka `DESC`)
- **Alter schema:** `ALTER TABLE book_inventory ADD page_count INT;`
- **Drop table:** `DROP TABLE table_name;` *(destructive)*

## CRUD operations (DML)

- **Create:**
    
    `INSERT INTO books (id,name,published_date,description) VALUES (1, 'Android Security Internals', '2014-10-14', '...');`
    
- **Read:**
    
    `SELECT * FROM books;` or `SELECT name, description FROM books;`
    
- **Update:**
    
    ```sql
    UPDATE books
    SET description = '...'
    WHERE id = 1;
    ```
    
- **Delete:**
    
    `DELETE FROM books WHERE id = 1;`
    

## Key clauses (refine queries)

- **DISTINCT:** remove duplicates → `SELECT DISTINCT name FROM books;`
- **GROUP BY:** aggregate by column(s) → `SELECT name, COUNT(*) FROM books GROUP BY name;`
- **ORDER BY:** sort → `... ORDER BY published_date ASC|DESC;`
- **HAVING:** filter *after* aggregation →
    
    `... GROUP BY name HAVING name LIKE '%Hack%';`
    

## Operators (filter logic)

- **Pattern:** `LIKE '%text%'` (often with `WHERE`)
- **Logical:** `AND`, `OR`, `NOT`
    - e.g., `WHERE category='Offensive Security' AND name='Bug Bounty Bootcamp';`
- **Range:** `BETWEEN a AND b`
- **Comparison:** `=`, `!=`, `<`, `>`, `<=`, `>=`
    - e.g., `WHERE published_date >= '2021-11-02';`

## Functions

**String**

- `CONCAT(a, b, ...)` → combine fields (alias with `AS`)
    
    `SELECT CONCAT(name,' is a type of ',category,' book.') AS book_info FROM books;`
    
- `GROUP_CONCAT(col SEPARATOR ', ')` with `GROUP BY` → list items per group
- `SUBSTRING(col, start, len)` → extract part of a string
- `LENGTH(col)` → character count (incl. spaces/punctuation)

**Aggregate**

- `COUNT(*)` → row count
- `SUM(col)` → total (non-NULLs)
- `MAX(col)` / `MIN(col)` → highest/lowest value
    
    (e.g., latest/earliest `published_date`)
    

## Quick mental checklist

- Start with the right DB: `USE <db>;`
- Create with correct types & constraints (`PRIMARY KEY`, `NOT NULL`, `AUTO_INCREMENT`).
- Always pair `UPDATE`/`DELETE` with a precise `WHERE`.
- Use `GROUP BY` + aggregates; use `HAVING` (not `WHERE`) to filter aggregated results.
- Prefer targeted `SELECT` columns over  in production queries.
- For deduping, start with `DISTINCT`; for counts per value, `GROUP BY name;`

That’s the essentials you’ll need to recognize schemas, write safe queries, spot SQLi patterns, and quickly sift data during investigations.
