# ZEEK

Zeek is a powerful, passive, open-source **Network Security Monitoring (NSM)** and traffic analysis framework. Unlike traditional IDS that focus solely on signatures, Zeek provides a rich ecosystem for deep packet analysis, event correlation, and forensic logging.

## 1. Core Architecture & Concepts

Zeek operates using two primary internal layers:

- **Event Engine:** Processes raw packets to describe events (e.g., a TCP connection or an HTTP request) without focusing on deep semantics. It handles protocol identification and file extraction.
- **Policy Script Interpreter:** Executes Zeek scripts to conduct semantic analysis and correlate events.

### Zeek vs. Snort

| Feature       | Zeek                                    | Snort                                            |
| ------------- | --------------------------------------- | ------------------------------------------------ |
| **Focus**     | Network analysis & metadata (NSM)       | Signature-based detection (IDS/IPS)              |
| **Mechanism** | Event-driven                            | Packet/Pattern-driven                            |
| **Strengths** | Complex threat detection, forensic logs | Speed, easy rule writing, blocking known attacks |
| **Cons**      | Steeper learning curve                  | Harder to detect complex, chained events         |

## 2. Operational Modes

- **Service Mode (`zeekctl`):** Used for live network monitoring.
- Commands: `start`, `stop`, `status`.
- Default Log Path: `/opt/zeek/logs/`.

- **Pcap Processing Mode:** Used for offline analysis.
- Command: `zeek -C -r sample.pcap`
- `-C` ignores checksum errors; `-r` reads the file. Logs are generated in the current working directory.

## 3. Log Management & Analysis

Zeek generates 50+ logs categorized into groups like **Network** (`conn.log`, `dns.log`), **Files** (`files.log`), and **Detection** (`intel.log`, `notice.log`).

### Key Analysis Tools

- **`zeek-cut`:** The primary tool for extracting specific columns from Zeek's tab-separated ASCII logs.
- _Example:_ `cat conn.log | zeek-cut uid id.orig_h id.resp_h`

- **CLI "Kung-Fu":** Standard Linux tools (`grep`, `sort`, `uniq -c`, `awk`) are essential for filtering and counting data.
- **UID Correlation:** Every session is assigned a unique **UID**, allowing analysts to pivot between different log files (e.g., finding the file hash in `files.log` and then finding the originating GET request in `http.log` using the same UID).

## 4. Detection Mechanisms

### Zeek Signatures (`.sig`)

Low-level pattern matching similar to Snort. They consist of:

1. **Conditions:** Header (IP/Port/Proto) and Content (Payload/Regex).
2. **Action:** Creates a record in `signatures.log` or triggers a script.

### Zeek Scripting (`.zeek`)

An event-driven language (e.g., `event zeek_init()`) used to automate tasks like:

- Extracting DHCP hostnames.
- Identifying brute-force attempts.
- Customizing logic for specific environments.

### Frameworks & Packages

- **File Framework:** Used for hashing (MD5/SHA256) and automatically extracting files from traffic into the `extract_files` folder.
- **Intelligence Framework:** Correlates traffic against threat intelligence feeds (e.g., `intel.log`).
- **Package Manager (`zkg`):** Installs third-party plugins like `geoip-conn` (for geolocation) or `sniffpass` (to detect cleartext passwords).

## 5. Summary Table of Vital Logs

| Log Name         | Purpose                                                  |
| ---------------- | -------------------------------------------------------- |
| **`conn.log`**   | IP, port, protocol, and duration for every connection.   |
| **`notice.log`** | Specific anomalies or signature matches.                 |
| **`files.log`**  | Metadata for files seen on the wire (MIME types, FUIDs). |
| **`intel.log`**  | Hits against defined Threat Intel indicators.            |
| **`weird.log`**  | Protocol violations and unexpected traffic behavior.     |
