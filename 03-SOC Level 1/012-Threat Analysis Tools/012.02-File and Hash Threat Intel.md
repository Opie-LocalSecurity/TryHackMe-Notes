# File and Hash Threat Intel

## **Purpose / workflow**

- SOC triage follows three steps: **verify → enrich → decide**. File & hash intelligence sit in **enrich**: turn a path/hash into contextual, actionable knowledge.

## Filepaths & filenames — quick heuristics

- **Filepath signals**
    - System drive (`C:\`) often used for persistence.
    - Shared/profile directories (`C:\Users\Public`, `Public Downloads`) enable cross-user access and evade strict per-user monitoring.
    - Temporary and writable system paths (`C:\Windows\Temp\`, `C:\ProgramData\`) are common staging/persistence locations.
- **Filename indicators of malice**
    - **Double extensions** (`invoice.pdf.exe`) exploit hidden extension settings.
    - **System binary impersonation** (`scvhost.exe`) — allowlist by path, not just name.
    - **High-entropy strings** (`jh8F21.exe`) — suggests packing/polymorphism.
    - **Masquerading / visual tricks** (`backup-2300.exe`, single-character swaps) to lower user suspicion.
- **Analyst guidance**
    - Treat names/paths as heuristics (not proof). Look for patterns and correlate with other telemetry.

## File-hash fundamentals

- **Why hashes matter**
    - Hashes (SHA256, MD5) uniquely identify binaries regardless of filename changes.
    - Always hash artifacts in the form encountered (zip vs extracted binary).
- **Commands**
    - Windows: `certutil -hashfile bl0gger.exe SHA256` or `Get-FileHash -Algorithm SHA256 bl0gger.exe`
    - Linux: `sha256sum bl0gger.exe`
- **Best practices**
    - Store hashes lowercase.
    - Include context (where/when seen).
    - Work with samples in isolated environments.

## VirusTotal — what to check

- **Key outputs to extract**
    - Detection score (vendor consensus).
    - Vendor threat labels and categories (family/capability).
    - Detection rules (YARA, heuristics, behavioural signatures).
    - Properties (file type, size, compile timestamp, entropy).
    - Contained domains/IPs and embedded files.
- **Triage heuristics / red flags**
    - ≥5 reputable vendors flagging is significant.
    - Conflicting vendor classifications — dig deeper.
    - Odd compile timestamps, high entropy (>7.5), or invalid/odd certificates.
    - Known-bad IPs or DGA-like domains in relations graph.
- **Analyst considerations**
    - New samples may have low initial detection — recheck after vendors update.
    - Cross-reference IPs/domains with other feeds (Shodan, etc.).
    - Signed binaries still may be malicious (stolen/abused certs).

## MalwareBazaar — cross-reference value

- **Features to use**
    - Sample searches (e.g., `sha256:<hash>`), family tags, downloadable samples.
    - YARA rules included with submissions — useful to add to EDR/SIEM hunting rules.
    - Campaign / threat-actor tags (e.g., `#TA551`) help attribution.
- **Operational use**
    - Use as a secondary source when VirusTotal detections are low or to obtain community YARA and samples for deeper analysis.

## Sandbox / dynamic analysis (SOC use)

- **Why detonate safely**
    - Confirm execution, extract runtime IOCs (domains, mutexes, dropped files), and map behaviours to MITRE ATT&CK.
    - L1: use sandbox lookups for quick context. Senior analysts: deep analysis (memory, syscalls).
- **Common tools**
    - **Hybrid Analysis** — behaviour trees, MITRE ATT&CK heatmap, fast executive summaries.
    - **Joe Sandbox** — deep system-call, memory, and strings analysis for reverse engineers and detection engineers.
- **What to extract from reports**
    - Threat score, ATT&CK techniques, network indicators, file drops, registry changes, injected processes.

## Sandbox limitations & caveats

- **Evasion**: environment checks, anti-debugging, hardware ID checks can cause false negatives.
- **Execution time**: many sandboxes run 2–5 minutes — multi-stage or delayed payloads may not show full behaviour.
- **Encrypted/obfuscated traffic**: TLS can't always be decrypted — C2 may be hidden; DNS tunnelling often missed.
- **Fileless / LotL techniques**: PowerShell, WMI, living-off-the-land tools may not produce meaningful disk artifacts and can evade sandbox detection.
- **Practical mitigation**
    - Combine sandbox results with other telemetry (endpoint logs, network captures, threat feed correlations).
    - Use YARA/rules from community repos and MalwareBazaar to supplement sandbox misses.
    - Treat a single sandbox report as one input — corroborate before declaring remediation actions.

## Practical outcomes / quick checklist for analysts

- Collect: filepath + filename + timestamp + SHA256 (lowercase).
- Query: VirusTotal → note detection score, labels, related IPs/domains.
- Cross-reference: MalwareBazaar for family tags and YARA.
- Detonate (if safe): Hybrid Analysis / Joe Sandbox for runtime IOCs and ATT&CK mapping.
- Correlate across telemetry and update EDR/SIEM rules (YARA/IOCs) where confidence is sufficient.
- Document provenance and context for every hash/sample to support remediation and hunting.
