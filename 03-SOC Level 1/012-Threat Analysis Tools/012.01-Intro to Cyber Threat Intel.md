# Intro to Cyber Threat Intel

## Why CTI matters for L1

- Cuts through alert noise by adding context so L1s can **triage fast and accurately**.
- Answers: **who/what is behind an indicator**, **past behavior**, **what to do now**.
- Early, informed decisions buy time for IR and reduce false positives.

## From data → information → intelligence

- **Data:** raw observable (e.g., `45.155.205.3:443`).
- **Information:** data + facts (e.g., WHOIS, first-seen date).
- **Intelligence:** analyzed info with **so-what** (e.g., “current BumbleBee C2 → block”).
- L1 responsibility: **enrich** artefacts until they’re actionable—or prove they’re not.

## Key terms

- **IOC:** evidence of compromise (e.g., C2 IP in logs).
- **IOA:** malicious action underway.
- **TTPs:** adversary methods mapped to **MITRE ATT&CK** IDs.

## Indicator types & first-look enrichment

- **IP:** WHOIS/ASN, VT relations, Shodan.
    
    IOA/TTP example: repeated SSH failures → `T1110.003`.
    
- **Domain/FQDN:** WHOIS age, passive DNS (RiskIQ/SecurityTrails), urlscan.io.
- **URL:** URLhaus, urlscan.io graphs, Any.Run (network off).
- **File hash:** VirusTotal, Hybrid-Analysis, MalShare.
- **Email addr:** MXToolbox headers, HIBP.
- **Local artefact (host/registry):** Sigma, EDR prevalence, vendor KBs.
    
    *Tip:* keep a **pre-filled lookup launcher** to save seconds per alert.
    

## Feeds vs Platforms

- **Feeds:** scheduled indicator streams (CSV/JSON/STIX/TAXII). Curate carefully to avoid FP fatigue.
- **Platforms:** structured stores tracking indicators, enrichment, relationships, permissions (e.g., **MISP**, **OpenCTI**).
    
    Practice: **ingest gradually**, measure actionability, promote to platform as **single source of truth**.
    

## Sources of CTI

- **Internal telemetry** (highest relevance): SIEM, EDR, phishing mailbox.
- **Commercial services:** premium feeds/sandboxes; high fidelity but licensing limits.
- **OSINT:** AbuseIPDB, URLhaus, public blogs/research—**cross-confirm** before action.
- **Communities/ISACs:** sector context with sharing labels.

## CTI classifications

- **Strategic:** high-level trends for business risk.
- **Tactical:** TTP-focused advisories.
- **Operational:** campaign-specific motives/intent.
- **Technical:** atomic indicators (IPs, hashes).
    
    L1s mainly handle **Technical**, observe **Tactical**, seed **Operational** insights.
    

## CTI lifecycle (Alex @ TryHatMe scenario)

### 1) Direction

- **Asset:** production PostgreSQL.
- **Risk:** GDPR fines, trust loss.
- **Controls:** NGFW (IP/domain blocking), EDR (hash quarantine).
- **Intel goals/questions:**
    - **Q1:** IPs/domains used to exploit/exfiltrate PostgreSQL now.
    - **Q2:** Active malware families targeting PgSQL drivers/creds + hashes.

### 2) Collection

- **Sources:** NGFW vendor feed; AbuseIPDB (PostgreSQL-bruteforce); internal **MISP**; current vendor threat report.
- **Hygiene:** export as STIX/CSV; store dated copies in “raw-intel” S3 for reproducibility.

### 3) Processing

- **Normalize** formats (IPv6 compression, lowercase domains, strip CIDRs).
- **Correlate/dedupe** against platform.
- **Tag** with source/date/**TLP**.
- **Convert** to actions: `firewall_blocklist.csv`, `edr_hash_rules.yar`.
    
    *Rule:* on conflicting TLPs, **apply the stricter label**.
    

### 4) Analysis

- Validate relevance before blocking:
    - **Q1:** Splunk shows NGFW-listed IP probed TCP/5432 → applicable.
    - **Q2:** OpenCTI links hash to *PgSteal*; Any.Run shows cred dumping; matches org’s ODBC driver.
- **Grading framework:**
    - **High:** ≥2 sources + local sighting → **block**.
    - **Medium:** one trusted source, no local hits → **alert**.
    - **Low:** OSINT-only → **monitor 14 days**.
        
        Outcome: **7 IPs + 1 hash** at High.
        

### 5) Dissemination

- **Firewall team:** CSV + change ticket (include **TLP** + risk rationale).
- **Endpoint team:** YARA rules via EDR console.
- **CTI platform:** full indicator objects w/ tags for history & correlations.
- **Management:** 200-word weekly risk summary (ROI/result focus).

### 6) Feedback (2 weeks later)

- **KPI improvements:**
    - Median dwell time for PgSQL brute-force IPs: **48h → 0h** (pre-emptive block).
    - **False positives:** 0% (none revoked).
- Next sprint: add **DNS tunneling IOCs** for same asset group.

## Sharing discipline

### TLP (FIRST.org)

- **CLEAR:** no restriction → wiki/platform.
- **GREEN:** share with peers, not public.
- **AMBER:** internal; external **need-to-know** only; reference in tickets.
- **RED:** named recipients only; avoid ticketing unless cleared.
    
    *Carry the TLP label with the IOC everywhere.*
    

### Intel formats & transport

- **STIX:** machine-readable JSON describing indicators/relationships/context.
- **TAXII:** secure APIs for near-real-time sharing (**Collections** and **Channels**).

## Supporting frameworks & vocab

- **MITRE ATT&CK:** standardize behaviors → cite technique IDs in tickets (e.g., `T1071.001`).
- **MITRE D3FEND:** defensive mappings; pick feasible mitigations (e.g., DNS request analysis for suspected tunneling).
- **Cyber Kill Chain:** phase model (Recon → Actions on Objectives) to locate/interrupt attacks.

## Vulnerability intel essentials

- **CVE:** identifier (e.g., CVE-YYYY-NNNN).
- **CVSS:** severity 0–10 with temporal/environmental modifiers.
- **NVD:** canonical link between CVEs, CVSS, affected products, exploits.

## Practical L1 takeaways

- Start every triage with **indicator type → right enrichment path**.
- **Automate** normalization/correlation and action-file generation.
- **Grade** indicators (High/Medium/Low) using **source agreement + local sightings**.
- **Tag & track** indicators in a **platform**; avoid feed sprawl.
- **Tailor dissemination** (right format, right audience) and measure outcomes with **KPIs**.
- Respect **TLP** and legal constraints; share only what’s necessary and allowed.
