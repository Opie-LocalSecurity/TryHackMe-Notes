# IP and Domain Threat Intel

## Why DNS matters in the SOC

- DNS is an early-warning dataset: suspicious domains surface in telemetry before file hashes.
- Analyst goal: turn a raw domain into contextual artefacts (ownership, resolving IPs, change frequency, behaviour vs CDN).

### Core DNS records for triage

- **A/AAAA** → current IPv4/IPv6. Rapid multi-ASN rotation = suspicious. Pivot A records to VirusTotal.
- **NS** → who runs the zone; recent/unusual providers can suggest fresh setup.
- **MX** → presence may support phishing use; note existence if web alert.
- **TXT** → SPF/DKIM policies; weak/absent SPF increases risk in mail cases.
- **SOA** → primary authority + serial; helps ownership picture.
- **TTL** → very low values (secs/mins) imply frequent changes.

### Common DNS abuse patterns

- **Fast flux**: many IPs + short TTLs; record churn and escalate.
- **CDN abuse**: changes within a single CDN ASN (Cloudflare/Akamai) are normal—verify reputation/ownership.
- **Typosquatting**: visual brand look-alikes → treat as high risk.
- **IDN homographs**: decode punycode (e.g., `xn--...`) and compare with brands.

### DNS triage workflow (L1)

1. **Snapshot**: collect A/AAAA, NS, MX, TXT, SOA, TTL (single view/screenshot or JSON).
2. **WHOIS check**: registrar, creation date, contacts.
3. **Interpret**: benign CDN vs throwaway domain (IPs, TTLs, ASN spread).
4. **Log evidence**: screenshots/JSON from DNS + reputation tools.
5. **Recommend**: block / monitor / close with rationale.

## IP Enrichment: Geolocation & ASN

### Purpose

- Add **RDAP ownership**, **ASN**, **geolocation**, **service context** to avoid over/under-reaction to raw IPs.

### RDAP (authoritative ownership)

- Provides **netrange**, **organisation**, **remarks (use type)**, **abuse contact**.
- Preserve **raw RDAP JSON** for audit; pivot to domains/certs if traffic is suspicious.

### ASN context & heuristics

- **Hosting ASNs**: many small netblocks, diverse tenants → common for malicious hosting.
- **Residential ISPs**: huge ranges → flags likely compromised consumer devices.
- **Cloud/CDN ASNs**: shared anycast edges → avoid broad blocks.
- Examples:
    - **AS32934 Meta** → social infra; issues likely account-level.
    - **AS16509 AWS** → massive cloud; scope to FQDN or tight CIDR, never entire ASN.
    - **AS124888 Vodafone** → residential/ISP; likely compromised endpoint.

### Geolocation (use cautiously)

- Country mismatches and city-level errors are common (CDN edges, cloud).
- Best practice: record **country from ≥2 sources**; treat as a hint, not deciding factor.

### IP enrichment workflow

1. **RDAP first**: netrange, org, ASN, abuse contacts.
2. **ASN role**: bgpview/ipinfo to classify (hosting/residential/cloud/CDN).
3. **Geo**: capture country from two sources; note discrepancies.
4. **rDNS**: note patterns (e.g., `.btcentralplus.com`); don’t rely solely on rDNS.
5. **Internal logs**: prior sightings in last 30 days and context.
6. **Classify**: decide role + reasoning.
7. **Plan outreach**: use abuse contacts if malicious and jurisdiction is cooperative.

## Service Exposure

### Why services/certs matter

- Open services reveal function and blast radius (e.g., exposed RDP on 3389 invites brute-force).

### Shodan/Censys reconnaissance

- Gather **open ports**, **service banners**, **software versions**, **cookies/markers**.
- Queries (e.g., `org:example.com`) map organisational exposure; version searches find vulnerable systems.

### TLS certificates & CT logs

- Inspect **issuer** (e.g., Let’s Encrypt vs self-signed), **validity period** (bursts/re-issues), **SANs**.
- **CT logs (crt.sh)**: enumerate issuance history; detect bursts/theme clusters.
- **Pivots**: reuse of certs across hosts, subject fragments, sibling hosts on Censys.

### Service exposure workflow

1. **Check banners** (Shodan/Censys): ports, products, versions, misconfigs.
2. **Review certs**: issuer, SANs, validity; capture interesting fields.
3. **Look for anomalies**: many unrelated SANs; rapid issuance bursts; brand look-alikes.
4. **Pivot**: from cert/banners to related infra.
5. **Assess blast radius**:
    - RDP/SSH on residential ASN → likely compromised endpoint.
    - TLS w/ many unrelated SANs on CDN ASN → shared infra; avoid IP blocks.
    - Self-signed TLS on small ranges → attacker panels/proxies likely.

## Reputation Checks & Passive DNS

### Why they matter

- Add **time context** to dynamic indicators (domains/IPs change tenants and use).

### Key sources

- **VirusTotal**: detection ratio, relations, first/last seen, community notes.
- **Cisco Talos**: web/email reputation, categories, volumes; vuln advisories + Snort rules.
- **IP2Proxy**: labels VPN/proxy/Tor exits (shared egress weakens attribution).

### Passive DNS signals

- **First/Last seen**, **IP churn**, **ASN spread** (multi-ASN = suspicious; single ASN often stable/CDN).
- Complement with **CT logs** (issuance bursts) and **Wayback** (content shifts).

### Reputation/history workflow

1. **VirusTotal**: record detections + timestamps.
2. **Talos**: score & category; note 30-day changes.
3. **IP2Proxy**: flag VPN/proxy/Tor.
4. **Passive DNS**: first/last seen, 7-day IP count, ASN spread.
5. **CT logs**: suspicious SANs / bursts.
6. **Wayback**: benign→phishing content switches.
7. **Decision**: block/monitor/close with expiry tied to activity.

## Operational Integration

### Safe enforcement patterns

- Prefer **hostnames** over IPs for CDN/anycast; use **DNS RPZ**, proxy categories, **SNI filtering**.
- For dedicated malicious VPS: block **/32** (narrow scope).
- **Expiry on blocks** (e.g., 7–14 days, auto-renew on re-observation).
- **Document in SOAR**: RDAP, screenshots, cert excerpts, and reasoning.

### Cautions

- **Geofencing**: raises priority but breaks workflows; use only with business-approved risk decisions.
- **Cloud/CDN pitfalls**: never block entire provider ranges (AWS/MS/Cloudflare). Act at domain/path; leverage provider abuse processes.

### Legal/provider considerations

- RIR ownership + **abuse contacts** inform takedown feasibility and evidence preservation.

### Playbook: Data → Decision

1. **Verify** indicator in your telemetry and relevance.
2. **Enrich**: DNS, RDAP, ASN, geo, services, certs, reputation, history.
3. **Score**: apply confidence matrix; record rationale.
4. **Decide**: block/monitor/allow; prefer precise controls; set expiry; document.
5. **Hunt & notify**: search for related IOCs; inform stakeholders; create follow-ups.

## Quick Reference — Red Flags & Tools

**Red flags**

- Very low **TTL** + frequent multi-ASN IP changes.
- **Young domain** with odd NS/MX/TXT patterns.
- **Self-signed** or burst-issued certs with many unrelated **SANs**.
- Exposed **RDP/SSH** on residential/hosting ASN.
- **VPN/Proxy/Tor** egress sources tied to phishing/c2.

**Go-to tools**

- DNS: nslookup.io, dnschecker.org
- Ownership: RDAP (ARIN/RIPE/APNIC), WHOIS
- ASN/Geo: bgpview.io, ipinfo.io, iplocation.net
- Services: Shodan, Censys
- Certs/History: crt.sh (CT), Wayback Machine
- Reputation: VirusTotal, Cisco Talos, IP2Proxy

This summary captures the key concepts, tools, workflows, and practical outcomes to support fast, safe triage by SOC L1 analysts.
