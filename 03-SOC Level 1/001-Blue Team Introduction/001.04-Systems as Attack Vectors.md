# Systems as Attack Vectors

## 🏰 **Definition of a System**

- A **system** refers to any platform storing or processing digital assets — e.g., physical servers, virtual machines, or cloud platforms (like Microsoft 365).
- **System value varies by target** — breaching a server affects many users, while a personal device impacts only one.

| System Example | Attack Value |
| --- | --- |
| Student laptop | Steal accounts, add to botnet |
| Bank admin laptop | Access internal banking systems |
| Law firm mail server | Exfiltrate all mailboxes, blackmail |
| Industrial control server | Deploy ransomware, halt operations |
| Gov website panel | Deface or spread disinformation |

**Core Idea:** Securing systems is critical because one breach can escalate from individual to organizational compromise.

## ⚔️ **Attacks on Systems**

### 1. **Human-Led Attacks**

- Users often **initiate compromise** unintentionally.
- Common causes:
    - Weak or reused passwords (→ 81% of breaches).
    - Malicious USBs or pirated downloads.
- Example: “RubberDucky” USB injects malware commands.

### 2. **Vulnerabilities**

- Software flaws enable exploitation.
- 2024: 40,000+ CVEs published; 300+ actively exploited.
- Admin missteps (weak passwords, open access) amplify risk.
- Example: thousands of outdated Windows XP/2008 systems still online.

### 3. **Supply Chain Attacks**

- Attackers compromise **trusted software components or updates**.
- Impact: All users of infected software are affected.
- Examples: **SolarWinds**, **3CX**, and even **TryHackMe’s Lottie Player** breach.
- Defense challenge: organizations cannot control all third-party code.

## 🧩 **Vulnerabilities**

### **Software Vulnerabilities**

- Exist in every application; may remain undiscovered for years.
- Example: **Shellshock (1992–2014)** in Linux.
- **Zero-day** = vulnerability unknown to the vendor, exploitable before a patch.

**Lifecycle:**

1. Vulnerability discovered → Assigned a **CVE** number.
2. Attackers race to exploit; defenders rush to patch.
3. Historical examples: **EternalBlue (2017)**, **PrintNightmare (2021)**, **Follina (2022)**.

### **Responding to Vulnerabilities**

- **Primary fix:** Apply vendor patches.
- For active/zero-day threats before patch release:
    - Restrict access (e.g., trusted IPs only).
    - Implement vendor-provided mitigations.
    - Block known attack patterns via IPS/WAF.

## ⚙️ **Misconfigurations**

### **Definition & Risks**

- Not software bugs — **human setup errors** (e.g., weak passwords, exposed databases).
- Common causes: convenience, oversight, or lack of security awareness.

**Real-world examples:**

- “123456” password exposed 64M McDonald’s job chat logs.
- Capital One AWS misconfiguration breached 106M customers.
- Smart fridges in botnets due to insecure IoT setups.

**Typical scenario:**

Weak password + Internet exposure → attacker breach within days.

### **Responding to Misconfigurations**

- No patch needed — just **secure configuration**.
- SOC/IT responses:
    - **Penetration Testing** – Ethical hacking to reveal weaknesses.
    - **Vulnerability Scans** – Identify default credentials or outdated software.
    - **Configuration Audits** – Manual checks using standards like **CIS Benchmarks**.

### 🔑 **Key Takeaways**

- Systems are core digital assets — protect them at infrastructure, software, and configuration levels.
- Most breaches start with human error or unpatched flaws.
- Ongoing defense involves **patch management, access control, monitoring, and configuration hygiene**.
