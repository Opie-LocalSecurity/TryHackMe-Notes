# Linux Logging for SOC

## **Context & scope**

- Focus: Linux server logging (Debian/Ubuntu/CentOS/RHEL) ‚Äî console/servers without GUI.
- Logs are plain text (mostly under `/var/log`) ‚Äî easy to read but less structured than Windows Event Logs.

## **Key log locations & meanings**

- `/var/log/syslog` (or `/var/log/messages` on some distros): aggregated system events (no strict schema).
- `/var/log/auth.log` (Debian/Ubuntu) or `/var/log/secure` (RHEL): authentication, sudo, user management, session open/close.
- `/var/log/kern.log`: kernel messages.
- Package manager logs: `/var/log/dpkg.log`, `/var/log/apt/` (Debian) or `/var/log/dnf.log`/`/var/log/yum.log` (RHEL).
- App-specific logs: e.g., `/var/log/nginx/access.log` for web requests.
- Per-user shell history: `~/.bash_history` (limited & easily evaded).

## **Practical commands & filtering**

- Use plain `cat`, `less`, text editors; filter with `grep` to reduce noise:
    - `cat /var/log/syslog | grep CRON`
    - `grep -R -E "auth|login|session" /var/log`
- For auth/session events:
    - `grep -E 'session opened|session closed' /var/log/auth.log`
    - `cat /var/log/auth.log | grep "sshd" | grep -E 'Accepted|Failed'`
- Bash history tools:
    - `cat ~/.bash_history` or `history` (note limitations and evasion techniques).

## **What to look for in `auth.log`**

- SSH success/fail patterns: `Failed password for X from <ip>` / `Accepted publickey for X`.
- Sudo usage and commands: lines containing `COMMAND=` show elevated commands executed.
- User management: `passwd`, `useradd`, `usermod`, `userdel` entries.

## **Limitations of standard logs**

- Low structure: no event codes; format varies by distro and configuration.
- Many runtime events (process creation, file changes, network calls) are **not** logged by default.
- `~/.bash_history` is unreliable ‚Äî attackers can evade it (leading space, scripts, alternate shells).

## **Runtime monitoring & system calls**

- Runtime visibility requires monitoring system calls (e.g., `execve`, `openat`) because user-space events map to kernel syscalls.
- Core idea: capture high-risk syscalls rather than everything (too noisy).

## **auditd (Audit Daemon)**

- Purpose: native Linux runtime auditing (rules in `/etc/audit/rules.d/`).
- Logs: `/var/log/audit/audit.log`; use `ausearch` for formatted queries.
- Useful `ausearch` example: `ausearch -i -k <key>` returns multi-line records split into PROCTITLE, CWD, PATH, SYSCALL.
- Important audit fields to use in triage:
    - `pid`, `ppid` ‚Äî process linking / process tree building
    - `auid` ‚Äî original login account
    - `uid` ‚Äî effective user running the action
    - `tty` ‚Äî session identifier
    - `exe` ‚Äî absolute path to binary executed
    - `key` ‚Äî rule tag for filtering
- Common uses: detect downloads (wget), edits to `/etc/ssh/sshd_config`, suspicious execs.
- **Tradeoffs with auditd**
    - Pros: built-in, syscall-level details, good forensic value.
    - Cons: verbose, heavy to ingest in SIEM if rules are overly broad.
- **Alternatives / complementary runtime tools**
    - **Sysmon for Linux** ‚Äî XML-style events; good if already using Sysmon on Windows.
    - **Falco** ‚Äî runtime security, good for containers.
    - **osquery** ‚Äî query-based host visibility.
    - EDRs ‚Äî vendor solutions providing varied runtime telemetry.
    - All these tools rely on system-call monitoring conceptually ‚Äî choose based on existing stack, ingestion needs, and noise tolerance.

## **SOC practical guidance**

- Start with `/var/log/auth.log` for immediate triage (logins, sudo, user changes).
- Use `grep` and focused searches to reduce noise; add auditd or equivalent for process/file/network runtime telemetry.
- Build balanced audit/syscall rules focusing on high-risk paths (binaries, config files, downloads, privilege escalation) to avoid data overload.
- Correlate audit fields (`pid/ppid`, `auid`, `exe`) with system logs and app logs (e.g., nginx) for full incident context.

## **Examples of high-value detections**

- Repeated SSH failures then an `Accepted` event from same IP.
- `sudo` lines showing `COMMAND=/usr/bin/systemctl stop edr` or `COMMAND=/usr/bin/su`.
- `ausearch` record showing `wget` followed by `execve` of downloaded artifact or edits to critical config files.
