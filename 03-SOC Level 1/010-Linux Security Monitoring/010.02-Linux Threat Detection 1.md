# Linux Threat Detection 1

## Key concepts

- **SSH is a primary Initial Access vector** on Linux: widely exposed (tens of millions of hosts), often misconfigured (password auth, weak passwords, leaked keys).
- **Two main SSH compromise paths:** stolen private keys (from code repos, automation servers, infected admin devices) and brute-forced or weak passwords.
- **Public services/webapps are another major entry point** (web servers, DBs, mail, Docker APIs, firewalls). Vulnerable apps lead directly to OS compromise (T1190).
- **Supply-chain and human-led attacks** are important: malicious packages, bad third-party actions, or admins copy-pasting unsafe scripts.

## Typical indicators & artifacts

- **SSH logs:** repeated failed attempts then `Accepted password` from external/unexpected IPs; `Accepted publickey` from internal/trusted IPs looks normal but still verify.
- **Web logs:** unusual query parameters (commands instead of IPs), 500 → 200 patterns around injected commands (e.g., `?host=;whoami`).
- **Process/activity evidence:** suspicious child processes launched by web apps (e.g., `curl … | sh`, `whoami`, `ls -la`, wget to suspicious hosts).
- **Threat intelligence / asset lookup:** IP reputation and historic login patterns help classify logins as benign vs malicious.

## Practical detection methods & tools

- **Log inspection:** start simple — list *successful* SSH logins and inspect username, auth method, source IP, and timing.
    - Example snippet used in notes:
        
        ```bash
        cat /var/log/auth.log | grep -E 'Accepted'
        ```
        
- **Web log analysis:** search access logs for suspicious query params and command-like inputs.
- **Process tree analysis (universal for service breaches):**
    - Use auditd / ausearch to trace an executed command back through parent PIDs to the originating service.
    - Example flow:
        
        ```bash
        ausearch -i -x whoami           # find whoami execution
        ausearch -i --pid <parentPID>  # walk up to parent / grandparent
        ausearch -i --ppid <pid> | grep 'proctitle'  # list children/commands
        ```
        
- **Correlate across sources:** SIEM alerts → process tree → web/app logs → TI lookups → developer/owner confirmation.

## Common real-world examples (lessons)

- **SSH misuse:** admin enables public SSH + password auth + weak password → brute force succeeds (e.g., Outlaw group campaigns).
- **Web app command injection:** an unsanitized `ping` endpoint allows `;whoami` → remote command execution, immediate OS risk.
- **Exposed management APIs (Docker, firewalls, email, WordPress plugins):** often exploited to drop shells or pivot.
- **Supply-chain incidents:** compromised build/actions or upstream libs can leak credentials or backdoor millions of systems.

## Detection workflow (recommended)

1. **Trigger**: SIEM alert or odd log entry (suspicious command, failed logins, odd web params).
2. **Contain & collect**: snapshot relevant logs, process lists, network connections, and affected files.
3. **Trace**: build a process tree from the suspicious command back to PID 1 using `ausearch` / audit logs.
4. **Correlate**: check web logs, SSH auth logs, asset/TI lookups for IP/user context.
5. **Decide**: determine if activity is admin/test vs malicious. If malicious — isolate host, rotate keys/passwords, and hunt for persistence.
6. **Remediate & review**: patch vuln, remove backdoors, rotate secrets, and improve controls (disable password auth, enforce key-based auth, limit public exposure).

## Practical mitigations (short)

- Disable SSH password authentication; use enforced key pairs and 2FA where possible.
- Limit SSH exposure (firewall, allowlists, jump hosts, bastions).
- Harden public services: patch promptly, minimize exposed management APIs, validate/filter user input.
- Monitor: centralize logs in SIEM, enable auditd for process tracing, and integrate TI for suspicious IPs.
- Supply-chain hygiene: lock down CI/CD, vet dependencies, and monitor code repo secrets.

## One-line takeaways

- SSH + public services = highest initial access risk on Linux; detection is efficient when you combine **simple log checks** with **process tree analysis** to root the activity to an application or user.
