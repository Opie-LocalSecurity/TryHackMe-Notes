# Linux Threat Detection 2

## Goal / big picture

- Most Linux breaches follow the same pattern: automated initial access by botnets → human operator takes over → manual follow-up (persistence, lateral movement, payloads).
- Detection focus: *early discovery activity* (commands and short sequences) and *Ingress Tool Transfer* (how attackers get binaries onto the host).

## Typical first-actions (Discovery)

- Attackers run the same forensic/diagnostic commands a sysadmin would; context matters.
- High-value early command to monitor: `whoami` — frequently executed by adversaries right after access.
- Common command categories and examples:
    - OS / filesystem: `pwd`, `ls /`, `env`, `uname -a`, `lsb_release -a`, `hostname`
    - Users / creds: `id`, `whoami`, `w`, `last`, `cat /etc/sudoers`, `cat /etc/passwd`
    - Processes / network: `ps aux`, `top`, `ip a`, `ip r`, `arp -a`, `ss -tnlp`, `netstat -tnlp`
    - Cloud / sandbox checks: `systemd-detect-virt`, `lsmod`, `uptime`, `pgrep "<edr-or-sandbox>"`

## Specialized discovery (goal-driven)

- Credential harvesting: `history | grep pass`, `find / -name .env`, `find /home -name id_rsa`
- Crypto-mining suitability: `cat /proc/cpuinfo`, `lscpu`, `free -m`, `top`
- Network scanning for new victims: `ping <ip>`, `for ip in 192.168.1.{1..254}; do nc -w 1 $ip 22; done`

## How attackers transfer tools (Ingress Tool Transfer)

- Common built-in tools abused to fetch payloads:
    - `wget URL -O /tmp/f`
    - `curl --output /var/www/html/backdoor.php URL`
    - `scp` / `sftp` (either attacker→victim or victim→attacker)
- Detection nuance: if attacker scp's in from their host you’ll see a new SSH session in the victim’s auth logs (but not a local exec of scp).

## Detection methods & telemetry to use

- **Process creation logging** (auditd / EDR): log executions of the command set above and `nohup`.
- **Audit/ausearch**: trace `whoami` → follow parent PID → build process tree for context.
- **Auth logs**: `/var/log/auth.log` — look for password-based SSH `Accepted` logins from external/untrusted IPs.
- **Network monitoring**: detect downloads from suspicious domains/IPs, SSH port-scanning from internal hosts, unusual outbound connections.
- **File activity**: new files in `/tmp`, `/var/tmp`, or oddly named hidden dirs; new archives like `dota3.tar.gz`.
- **EDR/AV**: alerts on known malware signatures (e.g., XMRig), suspicious persistent binaries.
- **Context rules**: differentiate admins/monitoring tooling vs unexpected processes (e.g., web server spawning `whoami` is suspicious).

## Process-tree hunting example (pattern)

1. `ausearch -i -x whoami` → find who executed it
2. `ausearch -i --pid <parent-pid>` → identify parent script/binary
3. `ausearch -i --ppid <parent-pid>` → list other commands spawned by that parent (reveal malicious script)

## “Hack-and-forget” attacker motivations

- Quick, automated gains at scale:
    - Install cryptominer (money)
    - Enroll host into botnet (DDoS, spam, proxying)
    - Use host as proxy for malicious traffic
- Attackers often change passwords / replace SSH keys to lock out owners and ensure persistence.

## Dota3 case study — key facts & indicators

- Initial access: large botnet (2k+ IPs across many countries) brute-forces SSH, targeting root and common weak passwords.
- Discovery: quick sequence of CPU/RAM queries (`cat /proc/cpuinfo`, `free -m`, `lscpu`) — strong signal of miner intent.
- Persistence steps observed: change user password, remove and replace `~/.ssh` with attacker key; comment `mdrfckr` is a unique fingerprint.
- Ingress & staging: attacker uploads `dota3.tar.gz` → extracts into hidden `/tmp/.X26-unix/...` and runs binaries from there.
- Payloads launched with `nohup` to survive session close:
    - `tsm` — custom SSH scanner for internal lateral discovery
    - `initall` — XMRig cryptominer
- Observable indicators:
    - auditd: creation of hidden directories in `/tmp`, extraction of `dota3.tar.gz`, execution of `nohup` binaries
    - network: internal SSH scanning across `192.168.*` and `172.16.*`
    - auth logs: suspicious SSH logins from external IPs
    - EDR/AV: detection of XMRig binary (VT example exists)

## Practical, prioritized detection & response actions (recommended)

1. **Alert on `whoami` executions** when parent process is a network-facing service (web server, sshd execs, cron from unexpected users).
2. **Log & alert**: password-based SSH `Accepted` from non-managed IP ranges or from IPs flagged by threat intel.
3. **Audit rules**: monitor execution of `wget`, `curl`, `scp`, `tar`, `nohup`, `chmod`, and suspicious `find`/`grep` patterns for secrets.
4. **File creation monitoring**: watch `/tmp`, `/var/tmp` for new hidden folders, new `.tar.gz` files, or filenames matching known malware.
5. **Process-tree correlation**: correlate process exec events with parent PIDs to separate admin automation from interactive attacker sessions.
6. **Network egress detection**: flag downloads from known malicious IPs/domains or accesses to public pastebin/GitHub payload URLs from servers that shouldn’t need them.
7. **EDR integration**: block common miner binaries and alert on suspicious long-running CPU-heavy processes started by non-root users.
8. **Hunt for `mdrfckr` and similar unique markers** in `~/.ssh/authorized_keys`, crontabs, and deployed scripts.

## One-page cheat-sheet (commands to hunt quickly)

- `ausearch -i -x whoami`
- `ausearch -i --pid <pid>` / `ausearch -i --ppid <ppid>`
- `cat /var/log/auth.log | grep "Accepted"`
- `ausearch -i -x wget` / `ausearch -i -x curl` / `ausearch -i -x scp`
- `ausearch -i -m PATH -f /tmp` (audit rule for /tmp file creation)
- `ss -tnlp` / `netstat -tnlp` to spot unexpected listening services
