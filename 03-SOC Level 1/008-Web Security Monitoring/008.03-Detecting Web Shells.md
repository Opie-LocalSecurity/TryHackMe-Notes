# Detecting Web Shells

## What it is

- **Web shell:** a malicious script uploaded to a web server that allows remote command execution (initial access and/or persistence).
- **Attack lifecycle:** used for reconnaissance, privilege escalation, lateral movement, credential theft, data exfiltration, and cleanup.

## How attackers place them

- **Root causes:** insecure file upload handling, misconfiguration, weak permissions, or prior compromise.
- **Common targets/paths:** upload directories (`/uploads/`, `/images/`, `/admin/`), default web roots (`/var/www/html`, `/usr/share/nginx/html`), temporary dirs (`/tmp`), platform-specific locations (e.g., `\inetpub\wwwroot\aspnet_client\` on Exchange).
- **Examples in the wild:** Hafnium and Conti exploited Exchange vulnerabilities to place `.aspx` web shells (including modifying existing files and adding backups).

## Anatomy / techniques used

- **Abuse of legitimate functions:** system execution calls (PHP: `shell_exec()`, `exec()`, `system()`, `passthru()`), `eval()` with obfuscated payloads, encoded query parameters (Base64, URL encoding).
- **Complexity range:** from single-line URL-exec payloads (`?cmd=whoami`) to full GUI shells with file managers and password protection.

## Log-based detection signals

- **HTTP request patterns**
    - Repeated GETs probing directories, followed by POST/PUT to upload locations.
    - Repeated GET/POSTs to the same file (possible shell interaction).
    - Unusual HTTP methods (`PUT`, `DELETE`, `OPTIONS`) used outside normal behavior.
- **Query strings**
    - Parameters like `cmd=`, `exec=`, abnormally long or encoded strings (Base64/URL-encoded).
- **User-Agent & IP**
    - Outdated, altered, or commonly blacklisted tools (`curl/…`, `wget/…`), unusual user agents, unexpected external IPs.
- **Referrer**
    - Missing referrer can be suspicious (but has many benign reasons).
- **Response codes & timing**
    - Track timestamps (off-hours activity), repeated 200/404 sequences during fuzzing.

## File system & host telemetry

- **File discovery**
    - Search for recently created/modified executable extensions: `.php`, `.jsp`, `.aspx`, double extensions (`image.jpg.php`).
    - Look for suspicious filenames or random strings in web directories.
- **Useful shell commands**
    - `find /var/www -type f -name "*.php" -newerct "YYYY-MM-DD" ! -newerct "YYYY-MM-DD"`
    - `grep -r "eval(" wp-content/`
- **Auditd**
    - Create/monitor rules for file creates/modifies and exec syscalls; search via `ausearch -k <key>` to correlate file events.

## Network / packet-level detection

- **PCAP indicators**
    - POST payloads containing source code of web shells, encoded commands, or command outputs.
    - Wireshark filters useful: `http.request.method == "METHOD"`, `http.request.uri contains ".php"`, `http.user_agent`.
- **Why packet capture helps:** validates payload (shows actual uploaded file content) and proves initial transfer.

## Correlation & tooling

- **Correlate logs:** link web server access/error logs with auditd events (creat/execve) to reconstruct attack sequence and identify actor/process.
- **SIEM benefit:** centralized collection, correlation rules (e.g., repeated POST → file creation → exec), targeted queries, alerting and faster triage.
- **Decoding / analysis tools:** CyberChef for Base64/URL decoding and payload inspection.

## Investigation & triage checklist (practical)

1. **Identify suspicious request(s):** look for unusual methods, `cmd`like params, encoded payloads, off-hours activity.
2. **Trace to filesystem:** check upload and web root directories for new/modified files around the timestamps.
3. **Confirm with host telemetry:** use auditd/`ausearch` for `creat`/`open`/`execve` events and UID/GID of actor.
4. **Capture network evidence:** export PCAPs of suspicious POSTs to inspect payloads.
5. **Contain:** remove or isolate identified web shell file(s), rotate credentials, block attacker IPs, patch vulnerability used for upload.
6. **Hunt for persistence & lateral movement:** search for additional web shells, new accounts, scheduled tasks, or unusual service changes.
7. **Improve detection:** add SIEM rules for suspicious sequences, tighten upload validation, harden permissions, and monitor for exec-related PHP functions.

## Recommended detection rules / signals to add (examples)

- Alert on POST/PUT to upload directories followed within X minutes by file creation events in the same dir.
- Alert on requests with query parameters containing `cmd`, `exec`, or long/encoded strings.
- Alert on requests with uncommon User-Agents or known scraping tools targeting upload endpoints.
- Periodic scans: `find` for newly created `.php/.aspx/.jsp` files and grep for `eval(`, `shell_exec`, `exec(`.
