# Detecting Web DDoS

## Core idea

- **DoS (Denial-of-Service):** single source overloads a service (app or OSI Layer 7 focus here) to make it unavailable.
- **DDoS (Distributed DoS):** many compromised devices (botnet) coordinate to overwhelm target, far higher scale and harder to block.

## Why attackers do it (motives)

- Financial loss, extortion (ransom-DDoS), hacktivism, distraction (to mask other intrusions), competitive sabotage, “denial of wallet” (force cloud costs), reputational damage.

## Common attack types (application / Layer 7 focus)

- **Slowloris:** many partial HTTP requests to tie up connections.
- **HTTP Flood:** high volume of valid/invalid HTTP requests.
- **Cache Bypass:** append random query strings to force hits to origin.
- **Oversized Query / Faulty Input:** force heavy server processing (e.g., huge `limit`).
- **Login/Form Abuse:** hammer authentication or password-reset endpoints.
- **Input validation abuse:** exploit poor input handling to crash/hang app.

## Typical targets (why these endpoints matter)

- `/login`, `/search`, `/api/*`, `/register`, `/cart` — endpoints that trigger DB queries, authentication, writes or expensive processing are high-value targets because each request is costly.

## Detection indicators (log / SIEM signals)

- **High request rate** from single IP or many IPs for same endpoint (e.g., 1,000 `GET /login`).
- **Burst timestamps:** dozens/hundreds of requests within seconds.
- **Odd or repeated User-Agents** (e.g., `curl`, `Python-urllib/3.x`) or same agent across many IPs.
- **Geographic anomalies:** globally distributed IPs inconsistent with normal user base.
- **Server errors spike (5xx)** — indicates resource exhaustion.
- **Logic abuse patterns:** repeated expensive query parameters (e.g., `limit=999999`).
- **Multiple layered signals** (high rate + odd UA + geography + 5xx) increase confidence.

Practical log/SIEM checks:

- Filter by `clientip`, `useragent`, `response` fields.
- Timechart request counts per endpoint.
- Pivot: top IPs → endpoints; top endpoints → response codes; UA → request rate.

## Forensic / investigation steps (hands-on)

- Open access logs and look for repeated identical or near-identical requests to heavy endpoints.
- Correlate timestamps to find bursts.
- Use SIEM to aggregate, chart, and pivot (e.g., Splunk `index="main"` then group by `clientip`, `useragent`, `uri`).
- Build watchlists for common indicators (UA strings, endpoints, abnormal geo ranges).

## Defenses & mitigations

### Application-level

- **Input validation**: enforce length/format limits, deny malformed or excessively costly queries.
- **Challenges**: CAPTCHAs or JS challenges to slow/stop bots.
- **Rate-limit logic**: per-IP / per-session limits for endpoints (e.g., `/login`).

### Network / infra

- **CDN**: cache static content at edge, absorb volumetric traffic, reduce origin load.
- **WAF**: block/challenge rules, rate limits, custom rules for targeted endpoints.
- **Load balancing & autoscaling**: distribute and absorb spikes where possible.
- **Large-scale mitigation providers**: cloud/CDN vendors can absorb huge attacks (industry examples show mitigation at massive scale).

### Operational

- **Instrument telemetry** (logs, metrics) for early detection.
- **Playbooks**: predefine actions (apply WAF rule, enable challenge, null-route, contact ISP/CDN).
- **Visibility**: use CDN/WAF dashboards to analyze origin vs edge traffic and spot cache bypass attempts.

## Common adversary bypass techniques

- Append random query parameters to bypass cache.
- Spoof/rotate User-Agent and referrer.
- Distribute requests widely across geographies and IP ranges to evade simple IP blocks.
- Vary request patterns to look more “human.”

## Quick analyst checklist (actionable)

- Identify spike: which minute/second did counts jump?
- Top endpoints during spike → prioritize `/login`, `/api`, `/search`.
- Top client IPs & geos → block or throttle obvious bots (but be careful with legitimate CDNs/proxies).
- Inspect UA strings and request patterns (bursts, identical params).
- Apply emergency WAF rule (rate limit/challenge for targeted endpoint).
- Engage CDN/ISP if volumetric; follow incident playbook and preserve logs for postmortem.

## Outcomes & best practice summary

- Combine **secure coding** + **WAF/CDN** + **SIEM monitoring** + **response playbooks** to reduce risk and response time.
- Detection relies on correlating multiple signals (rate, UA, geo, 5xx).
- Mitigation is layered: application hardening prevents low-effort attacks; CDN/WAF and provider support handle large, distributed campaigns.
