# Detecting Web Attacks

## 🧠 **Client-Side Attacks**

**Goal:** Exploit vulnerabilities in the user’s browser or behavior.

**Impact:** Theft of credentials, cookies, or sensitive data from user devices.

**Visibility Challenge:** SOC teams have limited insight—attacks run in the browser and often leave no logs or network traces visible from the server side.

**Common Attacks**

- **XSS (Cross-Site Scripting):** Injects malicious JavaScript into trusted sites. Can steal cookies or session tokens.
- **CSRF (Cross-Site Request Forgery):** Tricks browser into making unauthorized requests.
- **Clickjacking:** Overlays invisible elements to trick users into unintended clicks.

## 🖥️ **Server-Side Attacks**

**Goal:** Exploit vulnerabilities in web servers, app code, or backend systems.

**Focus:** Attacks backend logic, databases, or configuration.

**Defensive Advantage:**

Server activity generates logs and network traces—enabling monitoring and detection.

**Common Attacks**

- **Brute Force:** Automated password guessing to gain access (e.g., T-Mobile 2021 breach).
- **SQL Injection (SQLi):** Alters SQL queries to access or modify data (e.g., 2023 MOVEit breach).
- **Command Injection:** Executes OS-level commands via unvalidated input.

## 📜 **Log-Based Detection**

**Purpose:** Use web server logs to detect and trace attack activity.

**Access Log Key Fields**

1. **IP Address:** Identify anomalies or known bad IPs.
2. **Timestamp / Page:** Detect unusual access patterns.
3. **Status Code:** Track repeated errors (404, 403, 500).
4. **Response Size:** Outliers may indicate exploitation.
5. **Referrer / User-Agent:** Spot automated tools (e.g., `sqlmap`, `wpscan`).

**Example Attack Chain in Logs**

1. Directory fuzzing (`200` responses show valid paths).
2. Brute-force login (`POST /login.php` with multiple rapid attempts → `302 Found` indicates success).
3. SQL Injection payloads (`' OR '1'='1`) sent via `/search` form.

**Limitations:**

Access logs often exclude POST body data—visible indicators may be partial.

## 🌐 **Network-Based Detection**

**Purpose:** Examine raw packet data (e.g., via Wireshark) for deeper evidence.

**Visibility:** Reveals headers, payloads, credentials, and SQLi details (for unencrypted traffic).

**Example Attack Trace**

1. **Directory fuzz** → finds valid pages.
2. **Brute force** → successful login (`password123`).
3. **SQLi payload** → `' OR '1'='1` reveals database dump (user info in cleartext).

**Tools:** Wireshark filters (`http.user_agent`, destination IP) and “Follow HTTP Stream” for full session reconstruction.

## 🔰 **Web Application Firewall (WAF)**

**Purpose:** First line of defense—filters HTTP(S) requests before they reach the web server.

**Capability:** Decrypts and inspects traffic, applying rules to block or challenge malicious requests.

**Rule Types**

| Type | Function | Example |
| --- | --- | --- |
| **Attack Pattern Blocking** | Detects known payloads | Block `User-Agent: sqlmap` |
| **Source Blocking** | Uses IP reputation / geo-blocking | Block botnet IPs |
| **Custom Rules** | Tailored to app logic | Only allow `GET/POST` to `/login` |
| **Rate Limiting** | Prevents abuse | Limit 5 login attempts/min per IP |

**Challenge-Response (e.g., CAPTCHA):**

Used instead of outright blocking to verify human users—helps mitigate bot traffic (≈37% of web traffic).

**Threat Intelligence Integration:**

Modern WAFs auto-update with feeds covering:

- **OWASP Top 10 vulnerabilities**
- **Malicious IP lists** (botnets, anonymizers, VPNs)
- **Emerging CVEs and APT indicators**

## ✅ **Key Takeaways**

- **Client-side attacks** exploit users → hard to detect via SOC logs.
- **Server-side attacks** exploit backend → detectable via logs & network traces.
- **Log & packet analysis** are complementary for understanding full attack chains.
- **WAFs** are proactive defenses—filtering, blocking, or challenging threats before they reach the server.
