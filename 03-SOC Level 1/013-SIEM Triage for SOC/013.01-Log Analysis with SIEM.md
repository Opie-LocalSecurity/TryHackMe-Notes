# Log Analysis with SIEM

## Core Benefits

- **Centralisation**
    - Aggregate logs from hosts, network gear, identity, cloud, apps into one console.
    - Outcome: Faster triage and investigations (single pane vs. pivoting across IPS/EDR/etc.).
- **Correlation**
    - Link disparate events (e.g., IDS scan alert ↔ Sysmon/WinEvent details) to identify actor, host, tool, and intent.
    - Outcome: Turn fragments into an incident narrative; reduce false positives.
- **Historical analysis**
    - Query past activity to spot patterns (e.g., prior logins from same IP/geo).
    - Outcome: Distinguish benign anomalies from threats; support timeline building.
- **(Not covered in depth here):** Visualization, detection rules, automation (SOAR) further speed detection and response.

## Log Sources Overview

- **Host-based**
    - Workstations/servers (Windows, Linux; DNS/SQL/web servers).
    - Value: Nearly all attacks touch endpoints; rich context for processes, users, files.
- **Network-based**
    - Firewalls, routers, IDS/IPS.
    - Value: Traffic flows, lateral movement, exfil indicators; key for correlation.
- **Web-based**
    - Web server/application logs (Apache/Nginx/app logs).
    - Value: Detect brute force, scanning, exploitation, web shells, DDoS.
- **Also common (out of scope here):** Cloud (AWS/Azure), IdP (Entra ID), third-party SaaS/app logs.

### Time Pitfalls

- Logs arrive in mixed time zones (UTC vs. local; sometimes no TZ).
- Analyst local time may differ from SIEM’s normalized time.
- **Action:** Always confirm TZ normalization before drawing timeline conclusions.

### Log Normalisation

- Different native formats (JSON/XML/plain text) → normalized schema.
- **Action:** Use normalized fields for consistent search/correlation across sources.

## Windows Logs in SIEM

### Sysmon (must be deployed/configured)

- **What it gives:** Deep endpoint telemetry (process creation, network connects, registry, file, injection).
- **Key Event IDs**
    - `1` – Process Create
    - `3` – Network Connection
- **Use cases**
    - **Encoded PowerShell execution**
        - Example Splunk:
            
            ```bash
            index=winenv EventCode=1 *powershell* AND *EncodedCommand*
            | table _time ComputerName ParentUser ParentImage ParentCommandLine Image CommandLine
            ```
            
        - Outcome: Identify malicious PowerShell spawned by `cmd.exe` from suspicious JS.
    - **Suspicious outbound connection**
        - Example Splunk:
            
            ```bash
            index=winenv EventCode=3 ComputerName=WINHOST05
            | table _time ComputerName Image SourceIp SourcePort DestinationIp DestinationPort Protocol
            ```
            
        - Outcome: Attribute connection (proc, path, ports/IP); pivot to TI for IP reputation.

### Windows Event Logs

- **Security Log (high value)**
    - Detects auth events, account mgmt, object access, process creation (with auditing), policy changes.
    - **Key Event IDs**
        - `4720` – User account created
        - `4722` – User account enabled
    - Example:
        
        ```bash
        index=winenv EventCode=4720 OR EventCode=4722
        | table _time EventCode ComputerName Subject_Account_Name Target_Account_Name New_Account_Account_Name Keywords
        ```
        
    - Outcome: Spot persistence via rogue accounts.
- **System Log**
    - Service/driver state changes; good for persistence/priv-esc.
    - **Key Event IDs**
        - `7045` – Service installed
        - `7036` – Service started/stopped
    - Example:
        
        ```bash
        index=winenv EventCode=7045 OR EventCode=7036 ComputerName=WINHOST05
        | table _time EventCode ComputerName Service_Name Service_Account Service_File_Name Message
        ```
        
    - Outcome: Detect malicious service (e.g., runs from `%Temp%` under `SYSTEM`).

## Linux Logs in SIEM

### Authentication (`auth.log`)

- Tracks SSH logins, sudo/su, failures.
- **Brute force & anomalous logins**
    
    ```bash
    index=linux source="auth.log" *ubuntu* process=sshd
    | search "Accepted password" OR "Failed password"
    ```
    
    - Outcome: Count failed vs. accepted; escalate successful BF (actor gained access).
- **Privilege escalation indicators**
    
    ```bash
    index=linux source="auth.log" *su*
    | sort + _time
    ```
    
    - Outcome: Confirm root access obtained; pivot to other logs for method.

### System (`syslog`)

- Services, restarts, cron jobs—useful for persistence/exec cadence.
- **Cron-based persistence / reverse shell attempts**
    
    ```bash
    index=linux sourcetype=syslog ("CRON" OR "cron")
    | search ("python" OR "perl" OR "ruby" OR ".sh" OR "bash" OR "nc")
    ```
    
    - Outcome: Identify periodic scripts in `/tmp`, reverse shells (e.g., Perl to `10.10.101.12:9999`).

## Web Application / Server Logs

### Access Logs (Apache/Nginx/app)

- **WordPress brute force**
    
    ```bash
    index=* method=POST uri_path="/wp-login.php"
    | bin _time span=5m
    | stats values(referer_domain) as referer_domain values(status) as status values(useragent) as UserAgent values(uri_path) as uri_path count by clientip _time
    | where count > 25
    | table referer_domain clientip UserAgent uri_path count status
    ```
    
    - Outcome: Flag high-rate POSTs (e.g., Hydra UA) from source IPs.
- **Possible web shell activity**
    
    ```bash
    index=*
    | search status=200 AND uri_path IN(*.php, *.phtm, *.asp, *.aspx, *.jsp, *.exe) AND (method=POST AND method=GET)
    | stats values(status) as status values(useragent) as UserAgent values(method) as method values(uri) as uri values(clientip) as clientip count by referer_domain
    | where count > 2
    | table referer_domain count method status clientip UserAgent uri
    ```
    
    - Outcome: Surface suspicious script hits (e.g., `505.php`) with attacker fingerprints.
- **DDoS indicators**
    
    ```bash
    index=* status=503
    | bin _time span=10m
    | stats values(referer_domain) as referer_domain values(status) as status values(useragent) as UserAgent values(uri_path) as uri_path count by clientip _time
    | where count > 100000
    | table _time referer_domain clientip UserAgent uri_path count status
    ```
    
    - Outcome: Confirm overload (HTTP 503 spikes) and top offender IPs/paths/agents.

## Analyst Takeaways (Cheat Sheet)

- **Start broad → correlate:** Use network alerts to pivot into endpoint logs (Sysmon/WinEvent) for actor, tool, and scope.
- **Validate time zones:** Always align event times (local vs. SIEM UTC) before sequencing.
- **Favor normalized fields:** Consistent searches beat source-specific quirks.
- **Key Windows events:** Sysmon `1/3`; Security `4720/4722`; System `7045/7036`.
- **Linux focus:** `auth.log` for access & escalation; `syslog` for services/cron persistence.
- **Web quick wins:** High-rate POSTs to login endpoints; mixed GET/POST to scripts + `200`; 503 storms with six-figure counts.
- **Enrich with TI:** Check suspicious IPs/domains/process hashes from endpoint/network hits.
