# Alert Triage with Splunk

## 🧩 **Scenario 1: Initial Access (Linux Host Brute Force)**

**Alert:**

- *Brute Force Activity Detection*
- **Host:** `tryhackme-2404`
- **Source IP:** `10.10.242.248` (internal)
- **Time:** 17/09/2025 9:00:21 AM

**Investigation Steps:**

- Queried Splunk index `linux-alert` for events containing `"Accepted password"`, `"Failed password"`, or `"Invalid user"`.
- Observed:
    - High number of failed logins for non-existent users (indicative of enumeration).
    - One user (`john.smith`) with **503 login attempts**, showing a clear brute force pattern.
- Further query confirmed a **successful login** for `john.smith`, confirming credential compromise.

**Outcome:**

- **True Positive.**
- Attacker successfully brute-forced SSH access to the Linux host.
- **Action:** Escalate to SOC Level 2 / Incident Response for containment.

**Open Questions:**

- Why does the attacker have an internal IP?
- How were usernames enumerated?
- What did the attacker do post-compromise?

## 🧠 **Scenario 2: Persistence (Windows Scheduled Task)**

**Alert:**

- *Potential Task Scheduler Persistence Identified*
- **Host:** `WIN-H015` (workstation)
- **User:** `oliver.thompson` (System Engineer)
- **Task Name:** `AssessmentTaskOne`
- **Time:** 30/08/2025 10:06:07 AM

**Investigation Steps:**

- Queried Splunk index `win-alert` for `EventCode=4698` (scheduled task creation).
- Found a single event where:
    - Task runs daily on a workstation (unusual).
    - Executes `certutil` to download `rv.exe` from **tryhotme domain**, renames it to `DataCollector.exe`, then launches via PowerShell `Start-Process`.
    - Task runs under the compromised user `oliver.thompson`.

**Indicators:**

- Use of `certutil` + PowerShell = known attacker TTP for persistence.
- Custom domain suggests attacker-controlled infrastructure.

**Outcome:**

- **True Positive.**
- Confirmed malicious scheduled task for persistence.
- **Action:** Escalate to SOC Level 2 for deeper investigation and remediation.

**Open Questions:**

- How was the scheduled task created (manually or via script)?
- How was the user account compromised?
- What was the initial access vector to the workstation?

## 🌐 **Scenario 3: Possible Web Shell (Web Server Exploitation)**

**Alert:**

- *Potential Web Shell Upload Detected*
- **Website:** `http://web.trywinme.thm`
- **Suspicious IP:** `171.251.232.40` (Vietnam — flagged malicious >3000 times on AbuseIPDB)
- **Time:** 14/09/2025 9:31:51 AM

**Investigation Steps:**

- Queried Splunk index `web-alert` for activity from the IP.
- Found:
    - Large number of requests with **User-Agent: Hydra**, targeting `wp-login.php` → brute force attempt.
    - After filtering out Hydra activity, found **POST** request to `admin-ajax.php` with referer `theme-editor.php?file=b374k.php`.
    - Multiple successful **POST** requests to `b374k.php` afterward — evidence of active web shell use.
- Research confirmed `b374k.php` is a known PHP web shell.

**Outcome:**

- **True Positive.**
- Confirmed attacker brute-forced credentials, uploaded, and executed a known PHP web shell.
- **Action:** Escalate to SOC Level 2 / IR for server triage and cleanup.

**Open Questions:**

- Was the brute force attack successful in logging in?
- What upload mechanism was used?
- What commands/actions did the attacker execute through the shell?

## 🔎 **Overall Key Takeaways**

| Category | Indicators | Tools / Logs Used | SOC Action |
| --- | --- | --- | --- |
| **Initial Access** | SSH brute force from internal IP | Splunk (linux_secure logs) | Escalate; incident containment |
| **Persistence** | Suspicious scheduled task, `certutil`, PowerShell | Splunk (Windows Event 4698) | Confirm persistence; escalate |
| **Web Shell** | Brute force + PHP shell upload (`b374k.php`) | Splunk (web server logs) | Confirm compromise; escalate |

**Common Methodology:**

1. Validate alert details (host, time, user/IP).
2. Contextualize asset and user behavior.
3. Query logs for suspicious patterns or repeated activity.
4. Confirm malicious indicators.
5. Classify as **True Positive** or **False Positive**.
6. Escalate confirmed incidents for response.
