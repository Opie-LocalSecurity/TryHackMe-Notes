# Windows Logging for SOC

## Big picture

- **Logs = primary evidence.** OS/app events are recorded (EVTX files); Security logs are the highest-value source for SOC triage (authentication, user changes), but noisy and incomplete by default.
- **Goal:** detect intrusion *when* it starts (auth), *how* it progressed (processes/files/network), and *who* changed accounts (user management).

## Where to look / tools

- **Event Viewer** (`eventvwr`) — built-in GUI to read EVTX (left: sources, center: events, right: details/XML). Use **Filter Current Log** / **Find**.
- **Sysmon** (Sysinternals) — strongly recommended over raw 4688. Provides richer fields: command line, parent info, hashes, signatures. Sysmon events appear under `Applications & Services → Microsoft → Windows → Sysmon → Operational`.
- **VirusTotal** — check process/file hashes and IP/domain reputations.
- **Sysmon configs** — e.g. Florian’s / Neo23x0 config to tune what to log.

## Key Security Event IDs & how to use them

- **4624 — Successful Logon**
    - Use to detect suspicious RDP/network logins and entry points.
    - Core fields to review: **Logon ID**, **Logon Type**, **Account Name**, **Source IP**, **Workstation Name**, **Timestamp**.
    - Logon types: **3** (network) and **10** (RDP). NLA typically makes an initial type 3 precede type 10.
    - **Actionable:** save **Logon ID** to correlate subsequent events (processes, user changes).
- **4625 — Failed Logon**
    - Use to detect brute force / password spraying.
    - Filter for Logon Type **3** / **10** and look for repeated failures, common usernames (admin/helpdesk), unexpected source IPs.
    - Caveats: noisy and has many interpretation pitfalls.

## User management events (account lifecycle) — hunt for backdoors/persistence

- **4720** (created), **4722** (enabled), **4738** (changed) — user creation/enabling/modification.
- **4725** / **4726** — disabled / deleted accounts.
- **4723** / **4724** — password change / reset.
- **4732** / **4733** — member added/removed from a security group (e.g., Administrators).
- **How to triage:** identify **Subject** (actor) and **Object** (target), note **Logon ID** to link to the login session; red flags: unknown actors, out-of-hours changes, naming pattern mismatches, no IT approval.

## Process monitoring — Security Log 4688 vs Sysmon 1

- **4688 (process creation)**: available via Windows Security auditing but **noisy** and limited unless command-line auditing enabled.
- **Sysmon Event ID 1 (process creation)**: preferred — includes **PID**, **image path**, **command line**, **parent PID/path**, **hashes**, **signature**, **Logon ID**.
- **Triage steps (process launches)**:
    1. Filter Sysmon for **Event ID 1**.
    2. Check image path (unusual dirs: `C:\Temp`, `C:\Users\Public`), odd names (`aa.exe`, random chars), and hashes.
    3. Check parent process — unexpected parents (e.g., Notepad → CMD) are suspicious.
    4. Use **Logon ID** to tie processes back to the originating login session.
    5. If suspicious, map upward through parent chain and search for related events with same Logon ID.

## Sysmon — files, registry, network, DNS

- Useful event IDs: **11/13** (file create / registry value set), **3/22** (network connection / DNS query).
- Principle: these non-process events often lack Logon ID/parent info — tie them to the full context by matching **ProcessId** to the corresponding Sysmon Event ID 1.
- Red flags:
    - Outbound connections to external IPs on unusual ports (e.g., 4444) or known malicious IPs.
    - DNS queries to suspicious TLDs/domains.
    - Files written to staging dirs or persistence locations; scripts (`.ps1`, `.bat`) dropped.

## PowerShell monitoring

- **Problem:** a single `powershell.exe` process can run many commands without separate process creation events, so Sysmon alone misses command-level activity.
- **Practical, simple artifact:** **PSReadline history file** at
    
    `C:\Users\<USER>\AppData\Roaming\Microsoft\Windows\PowerShell\PSReadline\ConsoleHost_history.txt`
    
    - Logs every typed command when Enter is pressed, persists across reboots, created per user.
    - Limitations: records commands (not outputs), may not capture script contents executed via `.\script.ps1` or obfuscated/encoded payloads.
- Other advanced monitoring options (not covered in depth here): AMSI, transcription, constrained language, logging via Group Policy — each has tradeoffs.

## Quick practical checklist (SOC L1/L2)

- Filter Security log for **4624/4625** → look for unusual Logon Type, source IP, hostname, and repeated failures. Save **Logon ID** for correlation.
- Filter Sysmon for **Event ID 1** → check image/command line/parent/Hashes → query VirusTotal.
- For suspicious process → search Sysmon **ProcessId** across other Event IDs (file / registry / network / DNS).
- Search Security logs for **4720/4732/4724/etc.** to detect account creation/privilege changes; correlate by **Logon ID** and time.
- Check PSReadline history for typed PowerShell commands if PowerShell abuse is suspected.

## Recommendations / outcomes

- **Enable Sysmon** with a curated config (e.g., Florian/Neo23x0) for higher-quality telemetry.
- **Correlate by identifiers**: **Logon ID** (auth → user actions) and **ProcessId** (process → file/network changes). These are the glue for reconstructing attack chains.
- Use a combination of **Security logs + Sysmon + PowerShell history + threat intel checks (VirusTotal)** to move from detection → investigation → remediation.
