# Wireshark: Packet Operations

## Big-picture: What the **Statistics** menu gives you

- **Purpose:** Fast “scope scan” of a pcap: volumes, protocols, endpoints, conversations, and protocol-specific stats (e.g., DNS/HTTP).
- **Outcome:** Build/confirm a hypothesis quickly (what talked to whom, over what, how much, and with which errors/codes).

## Statistics menu — what to use & why

- **Resolved Addresses**
    - **What:** Lists IPs ↔ hostnames seen via DNS answers.
    - **Use:** Rapidly identify accessed resources; pivot to suspicious domains.
- **Protocol Hierarchy**
    - **What:** Tree of all protocols with counts/percentages.
    - **Use:** See dominant services/ports; right-click any node → **Apply as Filter** to focus the view.
- **Conversations**
    - **What:** 5 tabs (Ethernet, IPv4, IPv6, TCP, UDP) showing **talker↔listener** pairs with bytes/packets.
    - **Use:** Triage top talkers, long-lived flows, odd ports; double-click to filter a pair.
- **Endpoints**
    - **What:** Unique participants per layer (Ethernet/MAC, IPv4, IPv6, TCP, UDP).
    - **Use:** Inventory of assets involved; combine with **Name resolution** and **GeoIP** (if configured) for context.
    - **Tips:**
        - Enable vendor MAC resolution: **Endpoints → Name resolution** (lower-left).
        - Global name/port resolution: **Edit → Preferences → Name Resolution**.
        - GeoIP: add MaxMind DB path: **Edit → Preferences → Name Resolution → MaxMind database directories** (then IP details show GeoIP fields).
- **Protocol Details sub-stats**
    - **IPv4 / IPv6 Statistics:** Scope to one IP version to list related conversations/endpoints/errors.
    - **DNS:** Breakdowns by rcode/opcode/qtype/class; spot NXDOMAIN spikes, unusual types.
    - **HTTP:** Request/response code distribution; list of methods and original requests for quick IOC hunting.

## Packet filtering — principles you actually use

- **Two filter types (not interchangeable):**
    - **Capture filters** (BPF): pre-capture, byte/port oriented, fewer protocols. Example: `tcp port 80`.
    - **Display filters**: post-capture, protocol-aware (3k+ protocols). Example: `tcp.port == 80`.
- **When to use which**
    - Default: **capture broadly, filter via display** during analysis.
    - Use capture filters only when you’re confident the pattern won’t miss the event.

## Syntax quick refs

### Capture (BPF) — minimal set

- **Scopes:** `host`, `net`, `port`, `portrange`
- **Dirs:** `src`, `dst`, `src or dst`, `src and dst`
- **Protos:** `ether`, `wlan`, `ip`, `ip6`, `arp`, `tcp`, `udp`
- **Examples:**
    - `tcp port 80`
    - `host 10.10.10.111`
    - `net 10.10.10.0/24`
    - `udp src port 53`

### Display filters — essentials

- **Operators:** `==`, `!=`(deprecated style; prefer `!(expr)`), `>`, `<`, `>=`, `<=`
- **Logic:** `and`, `or`, `not` (or `!`)
- **Examples:**
    - IP: `ip.addr == 10.10.10.111` • `ip.src == 10.10.10.0/24`
    - TCP/UDP: `tcp.port == 80` • `udp.dstport == 5353`
    - HTTP/DNS: `http` • `http.request.method == "GET"` • `dns.flags.response == 0`
- **Toolbar behavior:** Autocomplete by protocol fields; color feedback
    - **Green** valid • **Red** invalid • **Yellow** works but unreliable.

## Common protocol filters you’ll reuse

- **IP layer**
    - `ip` — all IP traffic
    - `ip.addr == 10.10.10.111` — any direction
    - `ip.src == 10.10.10.111` / `ip.dst == 10.10.10.111` — direction-aware
    - `ip.addr == 10.10.10.0/24` — subnet match
- **Transport**
    - `tcp.port == 80` • `tcp.srcport == 1234` • `tcp.dstport == 80`
    - `udp.port == 53` • `udp.srcport == 1234` • `udp.dstport == 5353`
- **Application**
    - HTTP: `http`, `http.request.method == "POST"`, `http.response.code == 200`
    - DNS: `dns`, `dns.flags.response == 0` (queries), `dns.flags.response == 1` (answers), `dns.qry.type == 1` (A)

## Display Filter Expression helper

- **Where:** **Analyse → Display Filter Expression…**
- **Why:** Browse all protocol fields, expected types (int/string), and predefined enums; build filters without memorizing field names.

## Saving time: bookmarks, buttons, profiles

- **Filter bookmarks & buttons:** Save frequently used/complex display filters for 1-click reuse (toolbar icons).
- **Profiles:** Snapshot your whole Wireshark setup (colors, columns, filters, buttons) per use-case (e.g., **IR-Web**, **Malware-DNS**, **ICS**).
    - Manage via **Edit → Configuration Profiles** or status-bar **Profile** selector.

## Practical workflow (repeatable)

1. **Open pcap → Statistics**
    - Protocol Hierarchy → spot dominant protocols → right-click filter.
    - Conversations/Endpoints → top talkers, weird ports, one-to-many patterns.
2. **Name/Geo context**
    - Enable name/port resolution; load GeoIP DB (if allowed) → enrich suspects.
3. **Drill down with display filters**
    - Start broad (`ip.addr`, `tcp.port in {80 443}`) → refine with advanced ops (`contains`, `matches`).
4. **Pivot to protocol stats**
    - DNS/HTTP panes → response codes, query types, anomalous spikes.
5. **Codify**
    - Save useful filters as buttons; store the setup in a profile for the next case.

## Handy mini cheat-sheet

- **Beaconing guess:** `ip.addr == <victim> and frame.len < 150 and tcp.flags.syn == 0 and tcp.flags.fin == 0`
- **All failed DNS:** `dns.flags.response == 1 and dns.flags.rcode != 0`
- **Suspicious HTTP methods:** `http.request.method in {"PUT" "DELETE" "OPTIONS"}`
- **Non-standard web ports:** `tcp.port in {8080 8443 8000} and http`
- **Large DNS payloads (exfil?):** `dns and udp.length > 512`
- **TLS SNI contains domain:** `tls.handshake.extensions_server_name contains "example.com"`

Keep this summary beside Wireshark; combine **Statistics** for scoping with **Display Filters** for precision, then save what works into **buttons/profiles** so your next investigation starts at full speed.
