# Wireshark: Traffic Analysis

## Nmap scan detection

### **TCP flags quick filters**

- SYN: `tcp.flags == 2` or `tcp.flags.syn == 1`
- ACK: `tcp.flags == 16` or `tcp.flags.ack == 1`
- SYN,ACK: `tcp.flags == 18`
- RST: `tcp.flags == 4` | RST,ACK: `tcp.flags == 20`
- FIN: `tcp.flags == 1`

### **TCP Connect (`nmap -sT`)**

- Traits: completes 3-way handshake; often window size > 1024; works unprivileged.
- Open: `SYN → / ← SYN,ACK / → ACK` (then normal data)
- Closed: `SYN → / ← RST,ACK`
- Broad filter: `tcp.flags.syn==1 and tcp.flags.ack==0 and tcp.window_size > 1024`

### **SYN/“Half-open” (`nmap -sS`)**

- Traits: no full handshake; usually ≤ 1024 bytes; needs privileges.
- Open: `SYN → / ← SYN,ACK / → RST`
- Closed: `SYN → / ← RST,ACK`
- Broad filter: `tcp.flags.syn==1 and tcp.flags.ack==0 and tcp.window_size <= 1024`

### **UDP (`nmap -sU`)**

- Traits: no handshake; open often silent; closed → ICMP Port Unreachable.
- Closed filter: `icmp.type==3 and icmp.code==3` (ICMP dest unreachable, port unreachable)

## ARP poisoning / MITM

### **ARP essentials**

- L2 only, no auth, not routable. Look for Requests (opcode 1), Replies (opcode 2), gratuitous/announcements.

### **Fast filters**

- All ARP: `arp`
- Requests: `arp.opcode == 1` | Replies: `arp.opcode == 2`
- Gratuitous scan tell: `arp.dst.hw_mac==00:00:00:00:00:00`
- Duplicate/conflict: `arp.duplicate-address-detected or arp.duplicate-address-frame`
- Suspected scanner/flood from a MAC: `((arp)&&(arp.opcode==1))&&(arp.src.hw_mac==<mac>)`

### **What to confirm**

- Two different MACs claiming same IP (gateway likely victim): IP conflict warnings.
- Evidence of flood/scan: many ARP requests from one MAC across a range.
- Pivot to upper layers: add src/dst MAC columns; MITM shows victim’s IP traffic destined to attacker’s MAC.

## Host & user identification (DHCP, NBNS, Kerberos)

### **DHCP (`dhcp` or `bootp`)**

- Packet types: Request `dhcp.option.dhcp == 3`, ACK `== 5`, NAK `== 6`.
- Useful options:
    - Hostname (12): `dhcp.option.hostname contains "keyword"`
    - Requested IP (50), Lease (51), Client MAC (61)
    - Domain (ACK/15): `dhcp.option.domain_name contains "keyword"`

### **NetBIOS Name Service (NBNS)**

- Filter: `nbns`
- Hunt on names: `nbns.name contains "keyword"` (yields hostnames ↔ IPs, TTLs).

### **Kerberos**

- Filter: `kerberos`
- Users: `kerberos.CNameString and !(kerberos.CNameString contains "$")`
- Other handy: `kerberos.pvno == 5`, `kerberos.realm contains ".org"`, `kerberos.SNameString == "krbtg"`
- Requests may include client addresses/NetBIOS.

## Tunneling & C2 via ICMP / DNS

### **ICMP**

- Indicators: unusual volume, payload length, embedded higher-layer strings.
- Filters: `icmp` and payload length heuristic: `data.len > 64 and icmp`

### **DNS**

- Indicators: long subdomain queries, encoded strings, tool patterns (dnscat/dns2tcp), spikes per FQDN.
- Filters: `dns`
    - Known tools: `dns contains "dnscat"`
    - Long names (ignore mDNS): `dns.qry.name.len > 15 and !mdns`

## Cleartext protocols

### **FTP**

- Filter: `ftp`
- Response codes:
    - Info (1xx): e.g., 211/212/213; `ftp.response.code == 211`
    - Conn (2xx): e.g., 220 (ready), 227/228/229 (passive modes)
    - Auth (3xx/4xx/5xx): 230 (login), 331 (user OK), 530 (login failed)
- Commands:
    - User/pass: `ftp.request.command == "USER"` | `"PASS"`; specific pass: `ftp.request.arg == "password"`
- Brute-force tells:
    - `ftp.response.code == 530`
    - `(ftp.response.code == 530) and (ftp.response.arg contains "username")`
    - `(ftp.request.command == "PASS") and (ftp.request.arg == "password")`

### **HTTP / HTTP2**

- Filters: `http`, `http2`
- Methods: `http.request.method == "GET"` / `"POST"`; all: `http.request`
- Status codes: `http.response.code == 200|401|403|404|405|503` (pick as needed)
- Parameters:
    - User-Agent hunts: `http.user_agent contains "nmap|Nikto|Wfuzz|sqlmap"`
    - URLs/hosts/servers:
        - `http.request.uri contains "admin"` | `http.request.full_uri contains "admin"`
        - `http.host contains "keyword"` or equality; `http.server contains "apache"`
        - Content clues: `data-text-lines contains "keyword"`
- **User-Agent sanity**: look for multiple UAs from one host, typos (“Mozlila”), tool strings, payloads.

### **Log4Shell (Log4j) quick tell**

- POST heavy; look for JNDI/LDAP payloads or odd UA.
- Filters:
    - `http.request.method == "POST"`
    - `(frame contains "jndi") or (frame contains "Exploit")`
    - `(http.user_agent contains "$") or (http.user_agent contains "==")`

## Encrypted web traffic (HTTPS/TLS) — how to decrypt

### **TLS handshake discovery**

- Global: `tls`
- ClientHello + HTTP: `(http.request or tls.handshake.type == 1) and !(ssdp)`
- ServerHello + HTTP: `(http.request or tls.handshake.type == 2) and !(ssdp)`

### **Key log workflow**

- Enable browser SSL key logging (set `SSLKEYLOGFILE` env var in Chrome/Firefox) during capture.
- In Wireshark: **Edit → Preferences → Protocols → TLS** (or right-click a TLS packet) → add key log file.
- After loading keys, inspect decrypted views (Decrypted TLS, Reassembled SSL/TCP, HTTP2 headers).

## Bonus tools in Wireshark

### **Hunt cleartext credentials**

- **Tools → Credentials** (Wireshark ≥ 3.1). Extracts creds from FTP/HTTP/IMAP/POP/SMTP.
- Shows packet numbers for username and password—click to jump. Still verify manually.

### **Generate firewall rules**

- **Tools → Firewall ACL Rules** → outputs rule templates (iptables, Cisco IOS, ipfilter, ipfw, pf, Windows netsh) for outside interface deployment.

## Quick triage playbook

1. **Scope the scan:**
    - TCP: `tcp.flags.syn==1 and tcp.flags.ack==0` then split by window size (>1024 connect, ≤1024 SYN).
    - UDP closed hits: `icmp.type==3 and icmp.code==3`.
2. **Check L2 tampering:**
    - `arp.duplicate-address-*`, gratuitous ARP, per-MAC floods.
3. **Name the hosts/users:**
    - DHCP hostnames, NBNS names, Kerberos `CNameString` (exclude `$`).
4. **Hunt C2/exfil:**
    - `icmp` with `data.len > 64`; DNS long queries / tool strings.
5. **Mine cleartext:**
    - `ftp`, `http`, tool/user-agent keywords, credential view.
6. **If TLS hides it:**
    - Load SSL key log; pivot on ClientHello/ServerHello IPs; re-inspect decrypted HTTP/HTTP2.

These bullets capture the core concepts, key Wireshark filters, and what outcomes to expect (how each signal looks on-wire and how to prove it).
