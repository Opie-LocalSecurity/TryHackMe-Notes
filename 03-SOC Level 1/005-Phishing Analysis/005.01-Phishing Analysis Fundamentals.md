# Phishing Analysis Fundamentals

## **Overview**

- **Spam** and **phishing** are common *social engineering* attacks, with phishing typically using **email** as the attack vector.
- Even with layered security defenses, one user action (clicking a link or opening an attachment) can compromise a network.
- Security analysts must be able to **analyze, identify, and mitigate** malicious emails.

## **Email Address Structure**

- Invented by **Ray Tomlinson** for **ARPANET** (1970s).
- Format: `user@domain`
    - **User Mailbox** – recipient identifier (e.g., “billy”)
    - **@** – separator
    - **Domain** – mail server location (e.g., “johndoe.com”)
- Analogy: street (domain) and house number/name (user mailbox).

## **Email Delivery Protocols**

### **Primary Protocols**

- **SMTP (Simple Mail Transfer Protocol)** – sends outgoing emails (port 25).
- **POP3 (Post Office Protocol v3)** – downloads messages to one device; deletes from server unless “Keep on server” enabled.
- **IMAP (Internet Message Access Protocol)** – syncs messages across multiple devices; stored on server.

### **Email Transmission Process**

1. Sender composes and sends message.
2. SMTP queries **DNS** for recipient’s domain.
3. DNS responds with domain mail server info.
    
    4–6. SMTP transfers message through the Internet to destination mail server.
    
4. Message stored on POP3/IMAP server.
    
    8–9. Recipient’s client retrieves (POP3) or syncs (IMAP) email.
    

## **Email Components**

- **Header** – metadata (routing, sender, recipient, etc.).
- **Body** – plain text or HTML content.

### **Key Header Fields**

- **From**, **To**, **Subject**, **Date** – basic visible info.
- **X-Originating-IP** – sender’s source IP.
- **Smtp.mailfrom / header.from** – domains used in authentication.
- **Reply-To** – address replies are sent to (may differ from “From”).

Headers follow the **Internet Message Format (RFC 5322)** standard.

Security analysts use *raw headers* for in-depth tracing and authenticity checks.

## **Email Body**

- Can be **plain text** or **HTML** (enables styling, images, and links).
- HTML allows embedded **links** and **attachments**.
- Attachments have specific content headers:
    - **Content-Type:** file type (e.g., `application/pdf`)
    - **Content-Disposition:** inline/attachment
    - **Content-Transfer-Encoding:** often `base64` (can be decoded for analysis)
- Handle attachments cautiously—do **not** open directly.

## **Types of Malicious Emails**

| Type | Description |
| --- | --- |
| **Spam** | Bulk unsolicited emails (may include malware — *MalSpam*). |
| **Phishing** | Pretends to be a trusted entity to steal data/credentials. |
| **Spear Phishing** | Targets specific individuals or organizations. |
| **Whaling** | Targets executives or high-level staff (CEO/CFO). |
| **Smishing** | SMS-based phishing attacks. |
| **Vishing** | Voice-based phishing (phone calls). |

### **Common Phishing Traits**

- Spoofed sender name/domain.
- Urgent tone or alarming keywords (“Invoice”, “Suspended”).
- Mimics branding of trusted entities.
- Poor grammar or generic greetings.
- Malicious links (often shortened) or attachments.

### **Best Practice:**

*Defang* suspicious URLs/emails to make them unclickable before sharing (e.g.,

`http://www.badsite.com` → `hxxp[://]www[.]badsite[.]com`).

Tools: [**CyberChef Defang URL**](https://gchq.github.io/CyberChef/#recipe=Defang_URL).

## **7. Business Email Compromise (BEC)**

- **Definition:** An attacker gains access to an internal corporate account and uses it to deceive others into performing fraudulent actions (e.g., wire transfers).
- **Significance:** Common interview topic for cybersecurity roles.

## ✅ **Key Takeaways**

- Understand **email architecture** (SMTP, POP3, IMAP).
- Know how to **analyze email headers and bodies** for anomalies.
- Recognize **phishing techniques** and **defensive procedures**.
- Be aware of **BEC** as a major corporate threat.
