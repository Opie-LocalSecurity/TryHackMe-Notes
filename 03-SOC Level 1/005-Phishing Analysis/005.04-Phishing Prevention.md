# Phishing Prevention

## **Overview**

- **Phishing**: One of the most common initial attack vectors; attackers trick users into revealing information or credentials.
- **Defensive Focus**: Email authentication standards—**SPF**, **DKIM**, **DMARC**, and **S/MIME**—work together to verify sender legitimacy and message integrity.

## **Sender Policy Framework (SPF)**

- **Purpose**: Authenticates sending mail servers to prevent domain spoofing.
- **Mechanism**: The recipient’s mail server checks the sender’s IP against the domain’s SPF record in DNS.
- **Record Example**:
    
    ```
    v=spf1 ip4:127.0.0.1 include:_spf.google.com -all
    ```
    
    - `v=spf1`: Start of record
    - `ip4:`: Authorized sender IP
    - `include:`: Trusted domain’s SPF record
    - `all`: Reject unauthorized senders
- **Verification Outcomes**:
    - *Pass/Neutral/None*: Accept email
    - *SoftFail/PermError*: Flag as suspicious
    - *Fail/TempError*: Reject email
- **Tools**:
    - **SPF Surveyor** – Visualizes DNS SPF records
    - **Google Admin Toolbox (Messageheader)** – Analyzes email headers and SPF results (e.g., `softfail`)

## **DomainKeys Identified Mail (DKIM)**

- **Purpose**: Ensures message authenticity and integrity using cryptographic signatures.
- **Mechanism**:
    - Sender signs message with a **private key**.
    - Receiver verifies signature using a **public key** in DNS.
- **Record Example**:
    
    ```
    v=DKIM1; k=rsa; p=<public_key>
    ```
    
    - `v=`: DKIM version
    - `k=`: Key type (RSA)
    - `p=`: Public key for verification
- **Common Errors**:
    - `permerror`: Invalid or missing DKIM signature/record.
- **Tools**:
    - **DKIM Inspector / Validator** (Dmarcian) – Checks DKIM record health and syntax.

## **Domain-Based Message Authentication, Reporting, and Conformance (DMARC)**

- **Purpose**: Aligns SPF and DKIM authentication results and defines a policy for handling failures.
- **Mechanism**: Validates that the sender’s domain matches the SPF/DKIM-authenticated domain.
- **Record Example**:
    
    ```
    v=DMARC1; p=quarantine; rua=mailto:postmaster@website.com
    
    ```
    
    - `v=`: Version
    - `p=`: Policy (none, quarantine, reject)
    - `rua=`: Reporting address for aggregate reports
- **Tools**:
    - **Domain Checker (Dmarcian)** – Analyzes DMARC/SPF/DKIM setup (e.g., `p=reject` ensures strict enforcement).

## **Secure/Multipurpose Internet Mail Extensions (S/MIME)**

- **Purpose**: Adds encryption and digital signatures to emails for confidentiality and integrity.
- **Key Features**:
    - **Digital Signatures** – Authenticate sender, ensure non-repudiation, and verify integrity.
    - **Encryption** – Protects message confidentiality using public key cryptography.
- **Workflow**:
    - Sender signs message with private key.
    - Receiver verifies/decrypts with sender’s public key (from digital certificate).
    - Both exchange certificates for future secure communication.

## **Organizational Anti-Phishing Measures**

### **Technical Defenses**

- **Email Filtering**: Blocks based on IP/domain reputation.
- **Secure Email Gateways (SEGs)**: Detect spoofing and impersonation.
- **Link Rewriting**: Replaces links with safe versions for scanning.
- **Sandboxing**: Tests attachments/URLs in isolated environments.

### **User-Facing Controls**

- **Visual Indicators**: “External Sender,” “Suspicious Link,” etc.
- **Phishing Reporting Buttons**: Quick user reporting of suspicious emails.
- **Awareness Training**: Teaches recognition of phishing tactics.
- **Simulation Exercises**: Controlled phishing tests to reinforce learning.

## **Key Takeaways**

- **SPF**: Authorizes sending servers.
- **DKIM**: Ensures email integrity and authenticity.
- **DMARC**: Enforces domain alignment and handling policies.
- **S/MIME**: Provides end-to-end message encryption and signing.
- **Holistic Defense**: Combine technical controls and user training to reduce phishing risks effectively.
