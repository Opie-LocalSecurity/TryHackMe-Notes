# IDS Fundamentals

## 🛡️ Intrusion Detection System (IDS) Overview

- **Purpose:** Monitors network or host activity to detect malicious behavior that bypasses the firewall.
- **Analogy:**
    - **Firewall:** Gatekeeper (blocks unwanted connections).
    - **IDS:** Surveillance camera (monitors internal activity and reports anomalies).
- **Function:** Detects and alerts on suspicious traffic but does not block it.

## ⚙️ IDS Deployment Modes

1. **Host-Based IDS (HIDS):**
    - Installed on individual hosts.
    - Monitors local system activities (files, logs, processes).
    - Pros: Deep visibility of host behavior.
    - Cons: High resource usage, difficult to manage on large networks.
2. **Network-Based IDS (NIDS):**
    - Monitors network traffic for all connected hosts.
    - Provides centralized detection and visibility.
    - Commonly deployed at network choke points.

## 🔍 IDS Detection Modes

1. **Signature-Based IDS**
    - Detects threats using known attack patterns (“signatures”).
    - Effective for **known** threats.
    - Limitation: Cannot detect **zero-day** or novel attacks.
2. **Anomaly-Based IDS**
    - Learns normal network/system behavior and flags deviations.
    - Detects **unknown** or **zero-day** attacks.
    - Limitation: High false positives (can be reduced via tuning).
3. **Hybrid IDS**
    - Combines both methods to improve accuracy and coverage.
    - Uses signatures for known threats and anomaly detection for new ones.

## 🧩 Snort – Open-Source IDS

- **Developer:** Cisco (originally created in 1998).
- **Type:** Network Intrusion Detection System (NIDS).
- **Detection:** Uses both **signature-based** and **anomaly-based** methods via configurable rule files.
- **Customization:**
    - Built-in rule sets for known threats.
    - Allows custom rule creation and selective rule activation.

## 📦 Snort Modes

| Mode | Description | Typical Use |
| --- | --- | --- |
| **Packet Sniffer** | Displays packets without analysis. | Troubleshooting network performance. |
| **Packet Logging** | Logs traffic to PCAP for later analysis. | Forensics / attack investigation. |
| **NIDS Mode** | Real-time detection using rule sets. | Continuous security monitoring. |

## ⚙️ Snort Configuration

- **Main directory:** `/etc/snort/`
    
    Key files:
    
    - `snort.conf` → main config (network range, active rules).
    - `rules/` → rule files directory (e.g., `local.rules`).
- To monitor entire network: enable **promiscuous mode** on the interface.

## ✍️ Snort Rule Structure

**Example Rule:**

```
alert icmp any any -> $HOME_NET any (msg:"Ping Detected"; sid:10001; rev:1;)
```

**Components:**

- **Action:** `alert` → what Snort does upon a match.
- **Protocol:** e.g., ICMP, TCP, UDP.
- **Source/Destination IP and Port:** define traffic direction.
- **Message (msg):** alert description.
- **Signature ID (sid):** unique rule identifier.
- **Revision (rev):** rule version control.

## 🧪 Creating & Testing Custom Rules

1. Add custom rule in `/etc/snort/rules/local.rules`:
    
    ```
    alert icmp any any -> 127.0.0.1 any (msg:"Loopback Ping Detected"; sid:10003; rev:1;)
    ```
    
2. Run Snort in NIDS mode:
    
    ```bash
    sudo snort -q -l /var/log/snort -i lo -A console -c /etc/snort/snort.conf
    ```
    
3. Test with a ping:
    
    ```bash
    ping 127.0.0.1
    ```
    
    → Triggers “Loopback Ping Detected” alert.

## 🧾 Offline Analysis (PCAP Files)

- Snort can analyze saved network captures:
    
    ```bash
    sudo snort -q -l /var/log/snort -r sample.pcap -A console -c /etc/snort/snort.conf
    ```
    
- Useful for **forensic analysis** or **incident review**.

## ✅ Summary of Key Takeaways

- **IDS = Detection + Alerting** (no blocking).
- **HIDS vs. NIDS** = Host-level vs. Network-level monitoring.
- **Signature vs. Anomaly vs. Hybrid** = Trade-off between accuracy and adaptability.
- **Snort** = Flexible, open-source NIDS with customizable rules, supporting both live and offline analysis.
