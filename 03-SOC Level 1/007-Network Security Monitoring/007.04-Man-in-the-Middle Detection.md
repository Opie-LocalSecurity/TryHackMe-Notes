# Man-in-the-Middle Detection

## What it is

- **Man-in-the-Middle (MITM):** Attacker secretly intercepts and can alter traffic between two endpoints, enabling eavesdropping, credential theft, and content injection.

## How it works (two-step model)

- **Interception:** Insert into the path (ARP/DNS/IP spoofing, rogue AP).
- **Manipulation/Decryption:** Downgrade/strip encryption or inject/modify content (e.g., fake logins, altered responses).

## Common MITM techniques

- Packet sniffing on open or weakly protected networks
- Session hijacking via stolen cookies/tokens
- **SSL stripping:** HTTPS→HTTP downgrade
- **DNS spoofing:** Forged DNS answers/poisoned cache
- **IP spoofing** to masquerade as trusted host
- **Rogue Wi-Fi AP** to capture all client traffic

> ⚠️ Note: The Equifax 2017 breach is widely attributed to an Apache Struts vuln (CVE-2017-5638), not a MITM attack. Treat that example with caution.
> 

## Where MITM fits in the Cyber Kill Chain

- **Exploitation:** Abuses trust/weaknesses in ARP/DNS to seize the session path.
- **Installation/Delivery:** Once “in-middle,” injects malware/credential-stealers or persists via traffic manipulation.
- Detection here is high-value: it signals an active intrusion mid-chain.

## Detecting ARP Spoofing

### ARP quick recap

- **ARP** maps IP→MAC on a LAN. It’s unauthenticated, so **unsolicited `is-at`** replies can poison caches.

### Key indicators

- Duplicate IP↔MAC mappings
- Many unsolicited **ARP replies** (gratuitous ARP)
- Spikes in ARP volume
- Traffic unexpectedly routed via a new MAC (often “gateway” IP)
- Multiple destination MACs for the same gateway IP
- Probe/reply loops

### Wireshark filters used

```
arp                             # all ARP
arp.opcode == 1                 # ARP requests (who-has)
arp.opcode == 2                 # ARP replies (is-at)
arp.isgratuitous                # unsolicited replies
arp && arp.src.proto_ipv4 == 192.168.10.1 && eth.src == 02:aa:bb:cc:00:01
arp.opcode == 2 && arp.src.proto_ipv4 == 192.168.10.1
arp.opcode == 2 && _ws.col.info contains "192.168.10.1 is at"
arp.opcode == 2 && arp.src.proto_ipv4 == 192.168.10.1 && eth.src == 02:fe[REDACTED]
arp.duplicate-address-detected || arp.duplicate-address-frame
```

**Outcome:** Gateway IP **192.168.10.1** repeatedly advertised from a suspicious MAC (02:fe:…): confirmed ARP cache poisoning → attacker positioned between victim and gateway.

## Unmasking DNS Spoofing

### DNS quick recap

- **DNS** resolves domain→IP. **DNS spoofing** injects forged answers so victims connect to attacker-controlled IPs.

### Key indicators

- Multiple answers to the same query
- Replies from **unexpected IPs** (not a configured resolver like **8.8.8.8**)
- Very short **TTLs** (e.g., 1–30s)
- **Unsolicited** DNS responses (no matching request)

### Wireshark filters used

```
dns
dns.flags.response == 1 && ip.src == 8.8.8.8                     # legit reference
dns.flags.response == 1                                          # all responses
dns && dns.qry.name == "corp-login.acme-corp.local"
dns.flags.response == 1 && ip.src == 8.8.8.8 && dns.qry.name == "corp-login.acme-corp.local"
dns.flags.response == 1 && ip.src != 8.8.8.8 && dns.qry.name == "corp-login.acme-corp.local"
```

**Outcome:** Rogue DNS replies observed for **corp-login.acme-corp.local**, pointing victims to **attacker IP 192.168.10.55**.

## Spotting SSL Stripping

### What happens

- Attacker keeps **HTTPS↔server**, but downgrades **victim↔attacker** to **HTTP**, exposing credentials in plaintext.

### Indicators

- Initial HTTPS intent (443) followed by **HTTP (80)** to the same host
- Persistent **301/302** redirects to `http://` or link rewriting
- Possible TLS warnings/self-signed certs (not always present)

### Wireshark filters used

```
tls || ssl
tls.handshake.type == 1 && tls.handshake.extensions_server_name == "corp-login.acme-corp.local"
dns.flags.response == 1 && ip.src == 192.168.10.55 && dns.qry.name == "corp-login.acme-corp.local"
http && ip.src == 192.168.10.10 && ip.dst == 192.168.10.55
```

**Outcome:** After DNS spoof, **no victim→real-server TLS** handshakes; instead **HTTP** to attacker. Cleartext credential **POST** captured.

## End-to-end Attack Timeline (from the pcap)

1. **ARP spoofing:** Attacker poisons mapping for **192.168.10.1** → wins MITM position.
2. **DNS spoofing:** Victim query for `corp-login.acme-corp.local` answered by **192.168.10.55** (rogue).
3. **SSL stripping:** Victim connects over **HTTP** to attacker; credentials exfiltrated in cleartext.

## Practical takeaways (defender’s checklist)

- **LAN protections:** Dynamic ARP Inspection, DHCP snooping, port security; segment guest networks.
- **DNS hardening:** Use authenticated resolvers, DNSSEC validation, monitor for **multi-answer** anomalies and unexpected responders.
- **Web hardening:** Enforce **HSTS**, redirect `http→https` only; consider TLS cert pinning (where feasible).
- **Monitoring:**
    - ARP: duplicates, gratuitous spikes, gateway MAC changes
    - DNS: low-TTL forged answers, unsolicited responses
    - Web: HTTPS→HTTP downgrades, repeated 30x to `http://`
- **User/data safeguards:** MFA, password managers, avoid open Wi-Fi/force VPN.

This summary captures the concepts, tools (Wireshark + filters), methodology (traffic isolation → indicator hunting → pivoting), and outcomes (confirmed ARP/DNS spoofing, SSL stripping, credential theft).
