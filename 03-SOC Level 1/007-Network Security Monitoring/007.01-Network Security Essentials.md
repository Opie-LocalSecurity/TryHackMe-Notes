# Network Security Essentials

## Network Components (what they are + why they matter to security)

- **User Workstations (Endpoints)**
    - Entry point via phishing/malware; first signs may be C2 beacons in network logs.
    - Key evidence: process creation, persistence, outbound connections.
- **File & Database Servers**
    - Hold crown-jewel data; prime targets for ransomware and exfiltration.
    - Watch for mass file access, unusual backups, large outbound traffic.
- **Application Servers (Web, Email, VPN, etc.)**
    - Internet-facing → constant scanning for vulns/misconfig.
    - Monitor WAF/IDS/Firewall for: SQLi, XSS, traversal, brute-force, suspicious IPs.
- **Active Directory / Auth Services**
    - Identity backbone; compromise = domain-wide impact.
    - Monitor failed logons, odd hours/locations, privilege escalation, atypical resource access.
- **Routers & Switches**
    - Traffic fabric; compromise enables MITM/rerouting/backdoors.
    - Collect configs, flows, admin logins, change events.
- **Firewalls / Perimeter Devices**
    - Gatekeepers (allow/deny + deep inspection).
    - Earliest indicators: port scans, brute-force, exploit attempts (and full connection logs).

## Network Visibility (how we “see” attacks)

**Goal:** Correlate **host-centric** and **network-centric** logs to reconstruct timelines.

- **Host-Centric Logs** (per device)
    - OS logs (Windows Event Logs, syslog), application logs (web/db), security tools (AV/EDR/HIDS).
    - Use for: process/execution tracing, file/registry changes, logged-in users, forensic detail.
- **Network-Centric Logs** (between devices)
    - Firewalls, IDS/IPS, routers/switches (flows), web proxies, VPN concentrators.
    - Use for: who/when/where/how (edges, lateral movement, C2, exfil), broad scoping.

> Rule of thumb: Inside the room = host logs; in/out of the building = network logs. Correlate both.
> 

## Network Perimeter (what/why/how)

- **What:** Boundary controlling traffic between internal (trusted) and Internet (untrusted).
    
    Components: Firewalls, Routers/Gateways, **DMZ** (public-facing services), **VPN** gateways, cloud/virtual edges.
    
- **Why it matters:** First line of defense; first place you see probing. Weak perimeters → exposed RDP/SSH/DB, scanning, brute-force, exfil paths.
- **Analyst focus:**
    - Review firewall/WAF/IDS logs (blocked & allowed).
    - Spot scans, brute-force, exploitation, malware beaconing, unusual outbound.
    - Validate **what should be exposed** vs. what actually is.

## Monitoring the Perimeter — Patterns & Detections

- **Scanning**: one source → many ports/hosts → repeated BLOCKs/ALLOWs in short time.
- **Brute-force**: one source → one login endpoint → many FAILs.
- **Exploitation**: IDS/WAF signatures (SQLi/XSS/traversal/RDP/SMB).
- **Beaconing**: periodic, regular intervals to fixed external IP/port (e.g., :4444).
- **Exfiltration**: large/regular HTTP POSTs or atypical protocols/destinations.

## Hands-On Scenarios (what to look for)

### Scenario A: Port Scanning

- **Clues:** Same external IP hits many ports: 21, 22, 23, 25, 53, 445, 3389…
- **Verdict:** Recon; prioritize blocking/rating source, review exposure list.

### Scenario B: Web Attacks (WAF/IDS)

- **Clues:** `action=BLOCK`, signatures for **SQLi**, **XSS**, **Directory Traversal**.
- **Verdict:** Active targeting of web app; confirm app patch level & WAF efficacy.

### Scenario C: VPN Brute-Force

- **Clues:** Flood of `FAILED_AUTH` from a single IP; eventual `SUCCESS_AUTH`.
- **Action:** Lock offending source, enforce MFA, rotate creds, review assigned VPN IP for post-auth activity.

## Case Study: Initech Corp — Attack Path (from logs)

**Assets (key):**

- 10.0.0.50 = VPN-GW (Critical)
- 10.0.0.51 = Internal Web/App
- 10.0.0.20 = Finance server (SMB)
- 10.0.0.60 = Workstation
- 10.0.1.10 = DMZ-WEB

**Observed sequence (condensed):**

1. **External Recon**
    - Firewall shows repeated **BLOCKs** from a suspicious external IP probing internal ports (22/23/21/445/3389).
2. **Perimeter Foothold**
    - VPN logs: many FAILs against `svc_[redacted]` → **eventual SUCCESS**; attacker receives **10.8.0.x** address (initial access).
3. **Lateral Movement**
    - From the assigned VPN client IP: ALLOWs to internal hosts on **22/445/3389**; IDS flags **SSH scans**, **RDP brute force**, **SMB lateral movement**.
4. **C2 Establishment**
    - IDS: periodic **“Possible C2 Beaconing”** to external `:4444` at fixed intervals.
5. **Data Exfiltration**
    - Firewall + IDS: numerous **HTTP POST Large Upload** alerts from compromised internal IP to external **:80/:8080**.

**Bottom line:** Successful VPN credential abuse → internal scanning/exploitation (notably SMB) → persistent C2 → staged exfiltration. The perimeter devices detected each stage, but correlation/response lag allowed progression.

## Analyst Playbook (queries + actions)

**Quick CLI patterns (shown in walkthrough):**

- Find scanners: `grep "BLOCK" firewall.log | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -nr`
- Confirm allowed paths for the same IP: `grep "<IP>" firewall.log | grep "ALLOW"`
- VPN brute-force: `grep FAIL vpn_auth.log | awk '{print $3}' | sort | uniq -c | sort -nr`
- Trace post-compromise activity from assigned VPN IP: `grep "<VPN_IP>" firewall.log ids_alerts.log`
- Beaconing/exfil: search IDS for `C2` and `HTTP POST Large Upload`.

**Immediate containment steps:**

- Disable/rotate **compromised accounts**; enforce **MFA** on VPN.
- Block **malicious external IPs/ports** (`:4444`, exfil destinations).
- Quarantine **compromised endpoints**; collect **host forensics** (EDR sweep, memory, artifacts).
- Patch/lock down **SMB/RDP/SSH**; restrict to admin jump hosts; enable **network segmentation**.
- Tune **WAF/IDS** and firewall rules; add **rate limits** & geo/IP reputation.

**Hardening checklist:**

- Principle of least privilege (AD), unique admin creds, LAPS.
- Disable legacy protocols; enforce strong auth (Kerberos only, no NTLM where possible).
- Egress filtering (allowlist destinations/ports), proxy web egress, inspect TLS where permissible.
- Centralize logs in SIEM (e.g., Splunk index=`network_logs`) with correlation rules:
    - Multiple VPN failures → success → internal port scans within N minutes.
    - Beaconing periodicity detection (e.g., 60-min cadence to same IP/port).
    - Large HTTP POST volume from non-web servers.

## Key Takeaways

- You can’t defend what you can’t see: collect both **host** and **network** telemetry.
- **Perimeter** devices provide the earliest signals (scans/brute-force/exploits).
- **Correlation** across firewall, IDS/WAF, and VPN logs turns noise into an **attack timeline**.
- Watch for canonical patterns: **scan → credential abuse → lateral movement → C2 → exfiltration**.
- Pair detections with **rapid containment** and **long-term hardening** to prevent recurrence.
