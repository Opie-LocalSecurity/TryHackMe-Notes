# Snort

## What Snort is

- Open-source, rule-based NIDS/NIPS for live traffic analysis, packet logging, protocol analysis and real-time alerting.
- Developed/maintained by Martin Roesch, community contributors and Cisco Talos.
- Can run as a sniffer, packet logger, IDS, or inline IPS.

## Core components / operation logic

- **Packet Decoder** — captures & normalizes packets.
- **Preprocessors** — massage packets (reassembly, protocol heuristics) before detection.
- **Detection Engine** — applies rules/signatures to packets.
- **Logging & Alerting** — writes alerts/logs in configured formats.
- **Output Plugins** — export alerts/logs (file, syslog, DB, etc.).
- **DAQ modules** — packet I/O abstraction (pcap, afpacket, ipq, nfq, ipfw, dump).

## Modes of operation & common CLI patterns

- **Sniffer mode** (`v`, `d`, `e`, `X`, `i`) — tcpdump-like live packet viewing:
    - `v` verbose headers, `d` payload, `e` link-layer headers, `X` hex dump, `i <iface>` choose interface.
- **Packet logger mode** (`l`, `K ASCII`, `r`, `n`) — save pcap/tcpdump logs or ASCII categorized logs:
    - Default binary (pcap) logs readable by tcpdump/Wireshark; `K ASCII` creates human-readable per-IP folders/files.
    - `r` read/process existing pcap/log file; supports BPF filters (e.g. `'udp and port 53'`) and `n` to limit packets processed.
- **IDS/IPS mode** (`c`, `T`, `N`, `D`, `A`) — rule-driven detection/prevention:
    - `c <snort.conf>` specify config; `T` test config.
    - `N` disable logging; `D` daemonize; `A <mode>` alert output (console, cmg, fast, full, none).
    - **IPS inline**: use `Q --daq afpacket -i eth0:eth1` (requires at least two interfaces) to drop/reject packets.
- **PCAP investigation** (`r / --pcap-single`, `-pcap-list`, `-pcap-show`) — process single or multiple pcaps with rules; `-pcap-show` prints which pcap is being processed.

## Important CLI examples

- Test config: `sudo snort -c /etc/snort/snort.conf -T`
- Sniff interface `eth0` verbose: `sudo snort -v -i eth0`
- Packet logger in current dir: `sudo snort -dev -l .`
- ASCII logging: `sudo snort -dev -K ASCII -l .`
- Read pcap with rules, console alerts, limit 10: `sudo snort -c /etc/snort/snort.conf -q -r icmp-test.pcap -A console -n 10`
- IPS mode (drop): `sudo snort -c /etc/snort/snort.conf -q -Q --daq afpacket -i eth0:eth1 -A console`

## Snort configuration essentials (`snort.conf`)

- **HOME_NET / EXTERNAL_NET** — define protected vs external networks (scope of detection).
- **RULE_PATH / SO_RULE_PATH / PREPROC_RULE_PATH** — where rules and SO/plugin rules live.
- `#config daq` and `#config daq_mode` — select DAQ and inline mode (afpacket for inline).
- `# site specific rules` — include `$RULE_PATH/local.rules` and other rule files.
- Configure **output plugins** for desired alert/log formats.

## Rule structure (header + options)

- Basic header: `<action> <protocol> <src_ip> <src_port> <direction> <dst_ip> <dst_port> (options)`
    - Actions: `alert`, `log`, `drop`, `reject` (and others).
    - Protocol: `ip`, `tcp`, `udp`, `icmp` (Snort v2 rule keywords).
    - Direction operators: `>` (unidirectional) and `<>` (bidirectional).
- **General options**: `msg`, `sid` (unique rule id), `rev`, `reference`.
    - SID ranges: `<100` reserved, `100–999,999` shipped, `>=1,000,000` user rules (practical guidance: use high SIDs to avoid collisions).
- **Payload options**:
    - `content:"..."` (ASCII/HEX), `nocase`, `fast_pattern` (optimizes multiple content matches).
- **Non-payload options**:
    - `flags` (TCP flags), `dsize` (payload size), `id` (IP ID), `sameip` etc.

## Rule authoring tips / behavior

- Put local/user rules in `/etc/snort/rules/local.rules`.
- Use `fast_pattern` when multiple `content` options are present to speed matching.
- Test rules standalone (run Snort directly with the rules file) for debugging.
- Beware of false positives when using behavior-based models (need baselining).

## Rule sets & updates

- Rule types:
    - **Community rules** — free, public.
    - **Registered rules** — free but require registration (delay).
    - **Subscriber rules** — paid, frequent updates (official Snort feed).
- Keep `snort.conf` stable — change carefully; prefer incremental rule updates or management tools.

## Practical operational notes

- Running Snort requires root (creates log files owned by root); inspect logs with `sudo` or change ownership (`chown`) appropriately.
- `q` quiet suppresses banner output.
- `A` modes affect where/how alerts appear (console vs. files); `none` disables alerts but still logs traffic.
- Use BPF filters with `r` to limit processed packets when reading logs/pcaps.
- For IDS deployment, prioritize efficient rules and preprocessors to reduce CPU load and false positives.
