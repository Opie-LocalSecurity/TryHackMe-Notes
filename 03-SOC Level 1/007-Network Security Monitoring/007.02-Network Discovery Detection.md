# Network Discovery Detection

## **Network Discovery Overview**

- **Purpose:** Identify assets, IPs, ports, and services on a network.
- **Used by:**
    - **Attackers** → To find exploitable vulnerabilities (reconnaissance).
    - **Defenders** → To inventory assets, close unnecessary services, and patch vulnerabilities.

## **Detecting Network Discovery**

- **Challenge:** Both benign and malicious entities perform scanning (attackers, defenders, web crawlers, etc.).
- **SOC Mitigation Techniques:**
    - **Allowlist** known internal/external scanners to reduce false positives.
    - **Integrate Threat Intelligence (TI)** to flag scans from known bad IPs.
    - **Adjust alert severity** using TI while maintaining generic scan-detection rules.

## **External vs Internal Scanning**

| **Type** | **Source/Destination** | **MITRE ATT&CK Phase** | **Severity** | **SOC Response** |
| --- | --- | --- | --- | --- |
| **External Scan** | External → Org’s public assets | *Reconnaissance (TA0043)* | Low | Block offending IPs on perimeter firewall |
| **Internal Scan** | Internal → Internal IPs | *Discovery (TA0007)* | High | Escalate, investigate, perform root cause analysis |
- **External scan** = attacker not yet inside the network.
- **Internal scan** = attacker already has foothold → potential lateral movement.

## **Log Analysis for Scan Identification**

- **Tool:** SIEM-exported CSV logs (e.g., `log-session-1.csv`).
- **Commands:**
    - `head -n2 file.csv` → preview log structure.
    - `cut -d',' -f<columns>` → isolate source/destination IPs, ports, actions.
- Used to distinguish **internal vs external** scanning based on IP address ranges.

## **Horizontal vs Vertical Scanning**

| **Scan Type** | **Pattern** | **Purpose** | **Detection Indicator** |
| --- | --- | --- | --- |
| **Horizontal** | Same source IP → multiple destination IPs on **one port** | Identify hosts exposing a specific service (e.g., SMB port 445) | Multiple destination IPs, single destination port |
| **Vertical** | Same source IP → same destination IP across **multiple ports** | Identify open ports/services on one host | Same destination IP, multiple destination ports |
| **Mixed Scan** | Combination of both | Comprehensive network mapping | Both patterns appear in logs |

## **Mechanics of Scanning Techniques**

| **Technique** | **Description** | **Detection/Notes** |
| --- | --- | --- |
| **Ping Sweep (ICMP)** | Sends ICMP echo requests to find live hosts. | Often blocked by firewalls. |
| **TCP SYN Scan** | Sends SYN packet → SYN-ACK = port open; stealthy and common. | Harder to detect; mimics normal TCP behavior. |
| **UDP Scan** | Sends UDP packet; ICMP “port unreachable” = closed; silence = possibly open. | Slow and unreliable due to timeouts. |

## **Defender Considerations**

- Internal vulnerability or compliance scans should be **known, documented, and scheduled**.
- SOC should **exclude legitimate scanners** from detection rules to reduce alert noise.

## **Identifying Scans in SIEM (Kibana Example)**

- **Navigate:** Discover → Data View: *All logs* → *Search entire time range*.
- **Use:**
    - “+” to add fields as columns.
    - Filter controls to isolate specific IPs, ports, or event types.
- **Goal:** Identify scan type (horizontal/vertical) based on log patterns.

## ✅ **Key Takeaways**

- Both attackers and defenders perform network discovery; **intent** distinguishes them.
- **External scans** = low-severity reconnaissance; **Internal scans** = high-severity compromise indicator.
- **Scan patterns** (horizontal/vertical) reveal attacker objectives.
- **SOC analysts** must know scanning mechanics, differentiate authorized from malicious activity, and leverage SIEM tools for detection and investigation.
