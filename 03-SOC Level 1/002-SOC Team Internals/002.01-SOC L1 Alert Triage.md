# SOC L1 Alert Triage

## 🧩 **From Events to Alerts**

- **Events**: Any system activity (logins, downloads, process launches) recorded in logs by OS, firewalls, or cloud systems.
- **Alerts**: Notifications generated when specific event patterns match predefined detection rules in security tools (SIEM, EDR, etc.).
- **Purpose**: To highlight *suspicious or anomalous* activity so analysts don’t have to manually review millions of raw logs.

### **Alert Management Platforms**

| Type | Examples | Description |
| --- | --- | --- |
| **SIEM** | Splunk ES, Elastic | Centralized alert management; best for most SOCs. |
| **EDR/NDR** | MS Defender, CrowdStrike | Endpoint/network-based alerts; often integrated into SIEM/SOAR. |
| **SOAR** | Splunk SOAR, Cortex SOAR | Automates and aggregates alerts across multiple tools for larger SOCs. |
| **ITSM** | Jira, TheHive, Trello | Used for workflow/ticket management of alerts. |

## 🧠 **SOC Roles in Alert Triage**

| Role | Responsibility |
| --- | --- |
| **L1 Analyst** | First responder; reviews alerts, filters false positives, escalates real threats. |
| **L2 Analyst** | Performs deeper investigation and remediation of escalated alerts. |
| **SOC Engineer** | Ensures alerts contain necessary data for analysis. |
| **SOC Manager** | Oversees triage efficiency and ensures timely detection. |

## 📄 **Alert Properties**

| Property | Description | Example |
| --- | --- | --- |
| **Alert Time** | When the alert was generated (slightly after the actual event). | Event 15:32 → Alert 15:35 |
| **Alert Name** | Detection rule name summarizing the issue. | “Unusual Login Location” |
| **Severity** | Urgency level (Low → Critical). | 🟢 Low, 🟡 Medium, 🟠 High, 🔴 Critical |
| **Status** | Alert’s workflow state. | 🆕 New → 🔄 In Progress → ✅ Closed |
| **Verdict** | Classification of outcome. | 🔴 True Positive, 🟢 False Positive |
| **Assignee** | Analyst responsible for handling the alert. | “Assigned to: Analyst A” |
| **Description** | Rule logic, reasoning, and triage guidance. | Why/how the alert triggered |
| **Fields** | Contextual data triggering the alert. | Hostname, Commandline, IP, etc. |

## ⚖️ **Alert Prioritisation**

**Goal:** Decide which alerts to handle first.

**Typical order of operations:**

1. **Filter:** Skip alerts already being handled.
2. **Sort by Severity:** Tackle Critical → High → Medium → Low.
3. **Sort by Time:** Handle oldest alerts first to prevent delayed response to older incidents.

## 🕵️ **Alert Triage Process**

### **1. Initial Actions**

- Assign the alert to yourself and mark **In Progress**.
- Review alert details (name, description, key fields) before investigating.

### **2. Investigation**

- Identify **who** and **what** is affected (user, host, network, etc.).
- Determine **type of action** (phishing, malware, login anomaly, etc.).
- Check **related events** before/after the incident.
- Use **threat intelligence** and internal resources for validation.
- Reference **Workbooks/Playbooks** if available for standardized steps.

### **3. Final Actions**

- Decide verdict: **True Positive (malicious)** or **False Positive (benign)**.
- Document reasoning and steps in alert comments.
- Change alert **status to Closed**.
- (Escalation or additional reporting covered in later modules.)

## ✅ **Key Takeaways**

- Alerts are refined signals from raw security events.
- SIEM/EDR/SOAR tools centralize and manage alert workflows.
- L1 analysts are responsible for initial triage; accuracy and speed are critical.
- Prioritization ensures the most dangerous threats are handled first.
- Thorough investigation and clear documentation prevent missed attacks.
