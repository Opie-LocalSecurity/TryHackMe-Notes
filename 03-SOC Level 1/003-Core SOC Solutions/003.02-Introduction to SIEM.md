# Introduction to SIEM

## **Logs Everywhere**

### **Log Sources**

- **Devices generating logs:** endpoints (Windows/Linux), servers, firewalls, routers, etc.
- **Purpose:** record activities for security, troubleshooting, and forensics.

### **Types of Log Sources**

1. **Host-Centric Logs** – activities within a host
    - Examples: file access, authentication attempts, process executions, registry edits, PowerShell use.
2. **Network-Centric Logs** – communication between devices
    - Examples: SSH connections, FTP transfers, web traffic, VPN access, file sharing.

## **Challenges (“Answers Nowhere”)**

- **Numerous Log Sources:** hundreds of events per second across multiple systems.
- **No Centralization:** logs scattered, requiring manual collection via SSH/RDP.
- **Limited Context:** isolated logs seem benign; correlation reveals attacks (e.g., lateral movement).
- **Manual Analysis Limits:** volume too high for human-only review.
- **Format Variability:** each source uses different log formats.

## **Why SIEM (Security Information and Event Management)?**

### **Purpose**

Centralizes, normalizes, correlates, and analyzes logs to detect and alert on security incidents.

### **Core Features**

- **Centralized Log Collection:** gathers logs from all devices using agents or APIs.
- **Normalization & Parsing:** standardizes log format for consistent analysis.
- **Correlation:** connects events across systems to detect patterns (e.g., VPN login + PowerShell + data transfer → exfiltration).
- **Real-time Alerting:** triggers alerts based on detection rules.
- **Dashboards & Reporting:** visual summaries (e.g., failed logins, triggered rules, top domains).
- **Additional Capabilities:** integrates threat intelligence, data retention, advanced search.

## **Log Sources & Ingestion**

### **Common Log Types**

- **Windows:** Event Viewer (unique Event IDs).
- **Linux:** logs stored under `/var/log/` (e.g., `auth.log`, `cron`, `httpd`, `kern`).
- **Web Servers (Apache):** request/response logs for monitoring web traffic and attacks.

### **Ingestion Methods**

1. **Agent/Forwarder:** lightweight tool sending logs to SIEM (e.g., Splunk Forwarder).
2. **Syslog:** protocol for real-time log transmission.
3. **Manual Upload:** offline log ingestion for analysis.
4. **Port-Forwarding:** endpoints send data to SIEM via listening ports.

## **Alerting Process & Analysis**

### **Detection Rules**

- Logical conditions on normalized logs; trigger alerts when met.
- **Examples:**
    - 5 failed logins in 10s → *Multiple Failed Logins*
    - Successful login after failures → *Brute Force Success*
    - Event ID 104 → *Event Log Cleared*
    - Event ID 4688 + `whoami` → *Command Execution Detected*
    - Outbound traffic > threshold → *Data Exfiltration Attempt*

### **Alert Investigation Workflow**

1. **Review triggered alert** on dashboards.
2. **Validate rule conditions** and event context.
3. **Determine:**
    - False Positive → adjust rule.
    - True Positive → escalate and investigate.
4. **Possible actions:** contact asset owner, isolate host, block IP, enhance detection.

## **Key Takeaway**

Without SIEM, analyzing massive, diverse, and uncorrelated logs is inefficient and error-prone.

**SIEM streamlines detection, correlation, and response — turning scattered logs into actionable security intelligence.**
